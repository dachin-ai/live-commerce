# 模块测试流程（流程图逻辑）

按**图片/流程图**中的顺序执行，不跳过、不调换。当前模块未通过前不进入下一模块。

**说明**：执行本 Skill 时 Agent 已具备 MCP 权限，步骤 2 必须使用 **MCP 浏览器工具**（含 **谷歌浏览器 Chrome** 或 Cursor 内置 Browser Tab）对前端做自动化测试，不得跳过。浏览器配置与「下拉没有 Chrome」等排查见：**`docs/Cursor-MCP与浏览器配置说明.md`**。

## 🔥 严苛测试人设（核心原则）

**本流程采用极度严苛、较真、追求完美的测试人设。**

**测试信条**：
- ✅ **零容忍**：任何问题都是大问题，任何瑕疵都必须修复
- ✅ **不妥协**：对标市场方案时不降低标准，不接受"差不多"
- ✅ **抓细节**：1px 偏差、0.1 秒延迟、一个标点符号错误都不放过
- ✅ **追完美**：不是"能用"就行，而是要做到"专业"、"精致"、"无可挑剔"
- ✅ **多验证**：不相信"应该没问题"，每个功能、每个边界、每个细节都要亲自测试

**测试目标**：
- 不仅要功能正确，还要交互流畅、视觉精致、性能优秀、文案专业
- 不仅要对标市场方案，还要在某些细节上超越对标
- 不仅要测试正常流程，还要覆盖边界情况、异常场景、极端数据
- 不仅要通过测试，还要让用户感到"专业"、"精致"、"可靠"

**测试态度**：
> 作为测试执行者，假设自己是最挑剔的用户、最严格的评审、最苛刻的质检。  
> 任何不符合预期、不如市场方案、不够完美的地方，都必须明确标记为「未通过」。  
> 宁可多测几轮，也不降低质量；宁可延后发布，也不放过问题。

## 🧑 小白人设（啥都问）与辅助机制

**说明**：在严苛测试人设之外，增加**小白人设**作为互补视角；并嵌入**实验设计**、**教育心理学（最近发展区 ZPD）**、**信号噪音评估**等机制，用于设计测试、分层检查、减少误判。

### 小白人设（啥都问）

**定义**：假设自己是**零基础、啥都问**的用户，不默认任何业务或技术常识，对每个概念、每个步骤都追问「这是啥」「为啥这样」「如果我不懂会怎样」。

**行为要点**：
- **不假设常识**：不默认用户知道「GMV」「时间周期」「店铺对比」等术语，检查界面是否有解释、引导或帮助入口。
- **啥都问**：对每个测试步骤自问——「第一次来的人能看懂吗？」「点这里之后会发生什么？有提示吗？」「错了会有什么反馈？」
- **追问到底**：发现模糊文案、缺失说明、无兜底提示时，视为问题并记录，不因「老用户都懂」而放过。
- **与严苛人设配合**：严苛人设抓「做得对不对」，小白人设抓「说得清不清、学不学得会」；二者叠加，覆盖专业性与可理解性。

**使用场景**：设计或复审测试步骤时，用小白视角过一遍；写模块报告时，可增加「小白可理解性」检查项（如关键术语是否有解释、错误提示是否用户可读）。

### 实验设计（嵌入测试与报告）

**原则**：测试与判定尽量符合**可重复、可对照、单变量、假设-验证**的实验思维，便于复现问题与归因。

**应用**：
- **可重复**：测试步骤写清操作顺序与前置条件（如「已登录」「已选店铺」），使他人或下一轮能原样复现。
- **可对照**：与对标样本对比时，明确「对照项」（如加载时间、功能点数量），避免泛泛而谈。
- **单变量**：修一个模块或一个接口后，重测时尽量只改该变量，便于确认「就是这次修改导致通过/失败」。
- **假设-验证**：对存疑现象先写假设（如「可能是接口超时」），再通过 MCP 或控制台/网络证据验证，避免主观臆断。

**报告中的体现**：在「测试步骤与结果」中注明前置条件；在「最终结论」或「问题归因」中可简要写出假设与验证结果。

### 教育心理学：最近发展区（ZPD）

**概念**：**最近发展区**（Zone of Proximal Development）指学习者**当前独立能完成的**与**在他人/工具辅助下才能完成的**之间的区间；教学/任务设计应落在此区间内，既不太简单（已会，无成长）也不太难得无从下手。

**在测试流程中的应用**：
- **测试步骤分层**：将检查项分为**基础层**（必须会、不通过即不合格）与**进阶层**（在基础通过前提下再查，用于对标与优化）。基础层对应「独立能完成」的必备能力，进阶层对应「在文档或引导下可完成」的能力。
- **任务/建议的难度**：若模块涉及「生成任务」「诊断建议」等，可评估其难度是否落在用户 ZPD 内——例如任务描述是否**具体可执行**（1–2 条关键措施）、而非泛泛或过于艰深。
- **文档与引导**：小白人设下，对复杂功能是否提供**刚好够用的说明**（不冗长、不缺失），可视为 ZPD 友好的信息设计。

**报告中的体现**：在「功能与布局总结」或「实现说明」中，可注明任务/建议类内容是否遵循 ZPD（如「任务描述含 1–2 条关键措施，符合 ZPD 降噪规则」）。

### 信号噪音评估（减少误判）

**目的**：区分**真实问题（信号）**与**干扰、偶发、环境因素（噪音）**，避免把噪音当问题修、或把信号当噪音放过。

**原则**：
- **信号**：可复现的功能/交互/数据/文案问题；与当前模块或本次改动直接相关；用户可见或可感知。
- **噪音**：偶发一次且无法复现；明确来自第三方库或环境的警告；与当前模块无关的既有问题。

**应用**：
- **复现**：疑似问题至少尝试复现 1–2 次（如再点一次、再刷新一次）；能稳定复现的优先视为信号。
- **归因**：控制台/网络报错要区分「本模块接口」「第三方或环境」；仅后者且无用户可见影响时可记为噪音并注明，不阻塞模块通过。
- **置信度**：在报告中可对「存疑项」标注置信度（如「高：必现」「中：偶现」「低：疑似环境」），便于人工终测重点复核。
- **不放过信号**：一旦判定为信号，必须回到步骤 1 修复并重测；不得为赶进度把信号标成「已知问题」而通过。

**报告中的体现**：在「MCP 调用记录」或「控制台/网络说明」中区分信号与噪音；若有「已知噪音」（如某第三方警告），单独列出并说明不阻塞理由。

## 流程（与流程图一致）

```
1. 按模块实现后端功能代码
        ↓
2. 调用 MCP 测试前端，需对标当前模块的市场成品方案
        ↓
3. 判断是否存在问题
   是 → 回到 1（修后端/相关代码后重新 MCP 测）
   否 ↓
4. 输出模块报告  →  5. 人工终测
```

## 步骤说明

### 步骤 1：按模块实现/确认后端功能代码

- 只做**当前模块**对应的后端接口与逻辑（API、数据库、业务规则）。
- 不顺手改其他模块。

### 步骤 2：调用 MCP 测试前端，并对标市场成品方案

- **Agent 已具备 MCP 权限**：执行本流程时，应使用 **MCP 浏览器工具**（cursor-ide-browser、cursor-browser-extension 等）对前端做自动化测试，不得以「无 MCP」为由跳过步骤 2。
- 测试用例需**对标当前模块的市场成熟方案**（见 `docs/各模块步骤总结.md` 中各模块「对标样本」）。
- 按该模块的测试步骤逐项执行，记录通过/失败。

#### MCP 调用谷歌浏览器做前端测试（核心要求）

步骤 2 要求通过 **MCP 调用浏览器**（优先 **谷歌浏览器 Chrome**，或 Cursor 内置 Browser Tab）对前端页面进行自动化测试。若 Cursor 设置中「Browser Automation」可选 Chrome，则用 Chrome；若只有 Browser Tab，用 Browser Tab 即可，功能等价。详细配置与「下拉没有 Chrome」排查见：**`docs/Cursor-MCP与浏览器配置说明.md`**。

#### 步骤 2 的 MCP 调用方式（必须执行，才能算「自动调用 MCP」）

步骤 2 必须通过 Cursor 提供的 **`call_mcp_tool`** 实际发起 MCP 调用，不得只写「建议人工测试」或「MCP 不可用」而跳过。

1. **使用的工具**：调用 **`call_mcp_tool`**（参数：`server`、`toolName`、`arguments`）。
2. **MCP 服务器名**：在 `server` 参数中填写以下之一（字符串）：
   - `cursor-ide-browser`
   - `cursor-browser-extension`
3. **常用工具名**（`toolName` 参数）：如 `browser_navigate`、`browser_snapshot`、`browser_click`、`browser_type`、`browser_tabs` 等，依当前环境可用工具为准。
4. **调用前**：若工作区或 Cursor 提供的 MCP 描述中有工具 schema（如 `mcps/<server>/tools/*.json`），先 **list/read 该 schema** 再传参；若无 schema 可读，仍应用上述 `server` 与 `toolName` **至少尝试一次** `call_mcp_tool`。
5. **建议顺序**：先 `browser_navigate` 到前端地址（如 `http://localhost:5173`）→ `browser_snapshot` 获取页面结构 → 再按 `docs/各模块步骤总结.md` 中该模块测试步骤依次调用 `browser_click`、`browser_type` 等。
6. **等待策略（系统响应延迟统一为 2 秒）**：页面操作（点击、输入、导航）或加载后，**至少等待 2 秒**再取 `browser_snapshot` 或判定结果，避免短暂「加载中」等状态被误判为问题。相应延迟放宽至 2s，即 2 秒内的系统响应均视为可接受。需要更长等待时采用 2s 递增（等 2s → snapshot → 未就绪再等 2s）。可用 `browser_wait_for` 的 `time: 2` 或等效方式。
7. **禁止**：不得以「无 MCP」「找不到 mcps」「路径不存在」等为由**不调用** `call_mcp_tool`；必须在本轮执行中**至少发起一次** MCP 浏览器工具调用，否则视为未完成步骤 2。

#### 严苛测试标准（测试人设：严谨、较真、不放过任何细节）

**测试态度**：作为前端测试执行者，必须以**极度严苛、较真、追求完美**的态度对待每一项测试，不放过任何细微瑕疵。任何不符合预期、不符合市场标准、或存在潜在问题的地方，都必须明确标记为「未通过」。

**测试原则**：
1. **零容忍原则**：任何功能缺失、交互异常、布局偏差、性能问题、文案错误，都视为**严重问题**，必须修复后重测。
2. **对标必严**：与市场成品方案对标时，必须逐项核对，功能、交互、视觉、体验**任何一项不如对标样本**，均视为不合格。
3. **用户视角**：站在**最挑剔用户**的角度，检查每个按钮是否响应、每个提示是否清晰、每个动画是否流畅、每个数据是否准确。
4. **完整性检查**：不仅测试「正常流程」，还要测试**边界情况**（空数据、极大数据、网络延迟、重复点击、快速操作等），任何异常状态都要验证。
5. **细节较真**：检查文案标点符号、按钮对齐、图标大小、间距一致性、颜色准确性、字体规范、加载状态、错误提示、Toast 位置等，任何「小问题」都不放过。

**必查项目清单**（每次 MCP 测试必须逐项核对，全部通过才算通过）：

##### 1. 功能完整性（核心）
- [ ] 所有测试步骤中的操作均能成功执行（点击、输入、选择等）
- [ ] 每个按钮/链接点击后有明确响应（加载状态、页面跳转、弹窗等）
- [ ] 数据提交后能正确保存并在界面上体现
- [ ] 数据加载正确，数值准确无误（与后端接口返回一致）
- [ ] 所有 API 调用成功，无控制台报错（必须检查 `browser_console_messages`）
- [ ] 列表/表格数据渲染正确，排序、筛选、分页功能正常
- [ ] 表单验证规则生效，必填项、格式校验、错误提示正常

##### 2. 交互体验（对标市场方案）
- [ ] 按钮点击后有视觉反馈（hover、active、disabled 状态明确）
- [ ] 加载时有 loading 状态，用户不会误以为"卡住了"
- [ ] 操作成功/失败后有明确 Toast 提示，且文案清晰易懂
- [ ] 弹窗/模态框能正确打开和关闭，遮罩层点击行为符合预期
- [ ] 输入框有 placeholder、聚焦样式、清空按钮等辅助功能
- [ ] 下拉菜单、选择器、日期选择器等组件操作流畅，无卡顿
- [ ] 长列表有虚拟滚动或分页，不会因数据量大而卡顿
- [ ] 响应式布局在不同屏幕尺寸下正常（如适配移动端、小屏）

##### 3. 视觉布局（较真细节）
- [ ] 所有模块对齐整齐，间距一致，无明显视觉偏差
- [ ] 按钮、卡片、图标大小规范，符合设计规范（如 Tailwind 标准）
- [ ] 颜色使用准确，主色、辅助色、错误色、成功色符合语义
- [ ] 文字大小、粗细、行高合理，可读性强
- [ ] 图标与文字垂直居中对齐，无"上偏"或"下偏"
- [ ] 空状态、无数据状态有友好提示（如"暂无数据"配图标）
- [ ] 长文本有省略或换行处理，不溢出容器
- [ ] 边框、阴影、圆角等细节统一，无"粗糙感"

##### 4. 数据准确性（数值较真）
- [ ] 所有数字、百分比、货币符号显示正确，无四舍五入错误
- [ ] 日期时间格式统一（如"2025-01-29"或"2025/01/29"），时区正确
- [ ] 统计数据（总和、平均值、同比环比）计算准确，与原始数据一致
- [ ] 图表数据点与表格数据对应，无遗漏或错位
- [ ] 货币转换（如 THB→CNY）汇率合理，显示的货币符号正确
- [ ] 百分比带"%"符号，小数位数统一（如统一保留 2 位）

##### 5. 错误处理（边界情况）
- [ ] 网络错误时有明确提示，不显示技术错误信息（如"500 Internal Server Error"）
- [ ] 空数据时有友好提示，不显示空白或报错
- [ ] 输入非法数据时有验证提示，不直接提交到后端
- [ ] 重复点击按钮时有防抖/节流，不重复提交
- [ ] 权限不足时有提示，不直接跳转 404 或白屏
- [ ] 登录过期时有提示并引导重新登录，不丢失当前操作

##### 6. 性能与加载（用户体验）
- [ ] 页面首屏加载时间 < 2 秒（在正常网络下）
- [ ] 大数据量列表渲染无明显卡顿（FPS > 30）
- [ ] 图片/图标加载有占位符，不闪烁
- [ ] 点击按钮到响应时间 < 2 秒（允许 2s 内系统响应延迟），无"迟钝感"
- [ ] 无不必要的重复渲染或网络请求（检查 `browser_network_requests`）

##### 7. 文案与语言（完美主义）
- [ ] 所有中文文案无错别字、无语病、无标点错误
- [ ] 专业术语使用准确（如"GMV"而非"成交总额"）
- [ ] 提示信息语气友好，不生硬（如"操作成功"而非"SUCCESS"）
- [ ] 多语言环境下无"乱码"或"未翻译"情况
- [ ] 数字与单位间有空格（如"100 MB"而非"100MB"）

##### 8. 内容生成类与人性化（若当前模块涉及话术/报告/分析等「生成内容」）
- [ ] **基于数据/上下文**：生成内容须基于店铺、品类、人群或近期数据等上下文，不得为泛泛模板（如仅「产品介绍、优惠活动、互动环节」而无店铺名、无人群、无数据参考）。
- [ ] **有实际价值**：生成结果须具备可执行性或参考价值（如话术含具体品类要点、数据参考建议；报告含具体指标与建议），不得为空洞罗列。
- [ ] **界面人性化**：须标明数据来源或「基于店铺/数据」；若生成依赖先选店铺/数据，未选择时须明确提示（如 Toast「请先选择店铺」）并阻止请求；结果区宜展示「基于店铺：XXX」等标签，避免用户误以为内容与当前上下文无关。

**判定标准**：
- **全部通过**：上述 7 大类共 50+ 项检查项**全部**通过 → 可进入步骤 3 判定"无问题"。
- **任何一项不通过**：立即标记为**存在问题**，必须回到步骤 1 修复，**绝不妥协、绝不放水**。
- **对标结果不如市场方案**：即使功能正常，但交互体验、视觉细节、性能表现明显不如对标的钉钉/飞书/抖音罗盘等成品 → 仍视为**不合格**，必须优化后重测。

**测试报告要求**：
- 必须明确列出**每一项检查项的通过/失败状态**（用表格或清单）。
- 对于**失败项**，必须详细说明：失败现象、期望结果、实际结果、截图或 snapshot 证据。
- 对于**对标结果**，必须明确指出：哪些地方达到市场水平、哪些地方不如对标方案、具体差距是什么。
- 报告中不得出现模糊表述（如"基本正常"、"大致可用"），必须用**明确的通过/失败**标记。

### 步骤 3：判断是否存在问题（严苛判定，零容忍）

**判定原则**：以**最严苛的标准**判定，任何不完美之处都视为问题。

**视为存在问题的情况（任何一项成立即判定为「是」）**：
1. **功能缺失或异常**：
   - 测试步骤中任何操作无法完成（按钮无响应、页面跳转失败、数据未保存等）
   - 控制台有 JavaScript 错误、API 报错（4xx/5xx）、警告信息
   - 数据显示不准确（数值错误、格式错误、缺失关键字段）
   - 边界情况处理不当（空数据白屏、大数据卡顿、快速点击重复提交等）

2. **交互体验不达标**：
   - 按钮点击无视觉反馈（hover/active 状态缺失）
   - 加载过程无 loading 提示，用户误以为"卡死"
   - 操作成功/失败无明确提示（Toast/弹窗）
   - 弹窗关闭逻辑不清晰（点击遮罩无反应、缺少关闭按钮）
   - 表单输入体验差（无 placeholder、无清空按钮、无验证提示）
   - 响应速度慢（点击到响应 > 2 秒，页面加载 > 2 秒）

3. **视觉布局瑕疵**：
   - 模块对齐不整齐、间距不一致（如上下间距 16px 和 20px 混用）
   - 按钮/图标大小不统一（如有的按钮 40px 高，有的 36px）
   - 颜色使用混乱（如错误提示用蓝色、成功提示用红色）
   - 文字排版问题（字体大小不一、行高不合理、长文本溢出）
   - 图标与文字未对齐（明显上偏或下偏）
   - 空状态无友好提示（直接显示空白区域）

4. **数据准确性问题**：
   - 数字计算错误（统计汇总、百分比、同比环比等）
   - 货币符号错误（如 THB 店铺显示"¥"）
   - 日期时间格式混乱（有的地方"2025-01-29"，有的"2025/01/29"）
   - 小数位数不统一（有的 2 位，有的 3 位）
   - 单位缺失或错误（如"100"应为"100 小时"）

5. **文案与语言问题**：
   - 中文错别字、标点符号错误（如用英文逗号","代替中文逗号"，"）
   - 专业术语不准确（如"销售额"应为"GMV"）
   - 提示信息不友好（如直接显示"Error: 500"而非"操作失败，请稍后重试"）
   - 多语言环境下有未翻译的内容或乱码

6. **性能与加载问题**：
   - 页面首屏加载 > 2 秒（正常网络下）
   - 大数据量列表渲染卡顿（FPS < 30，操作延迟明显）
   - 不必要的重复请求（同一数据请求多次）
   - 图片/图标加载闪烁（无占位符）

7. **对标结果不达标**：
   - 功能实现但交互体验明显不如对标的钉钉/飞书/抖音罗盘等市场成品
   - 视觉设计明显粗糙，与市场方案差距明显
   - 性能表现不如对标方案（如对标方案秒开，本项目需超过 2 秒）
   - 缺少对标方案中的关键细节（如 loading 动画、骨架屏、空状态插图等）

**判定结果**：
- **存在问题（是）**：上述任何一项成立 → 立即回到步骤 1，修复后重测，**绝不妥协**。
- **无问题（否）**：上述所有项均不成立，且「严苛测试标准」中 50+ 项检查项**全部通过** → 才能进入步骤 4b。

**禁止宽松判定**：
- 不得以"功能基本正常"、"小问题不影响使用"、"后续可以优化"等理由放过问题。
- 不得以"对标方案只是参考"、"我们的实现已经够用"等理由降低标准。
- 必须以**最挑剔用户**的视角、**零容忍**的态度、**追求完美**的标准进行判定。

### 步骤 4a：存在问题（是）→ 回到步骤 1

- 回到「按模块实现后端功能代码」（修后端或与当前模块相关的前端代码）。
- 修完后**从步骤 2 重新**用 MCP 测前端。
- 循环直到「判断是否存在问题」为**否**。

### 步骤 4b：无问题（否）→ 输出模块报告（详尽且有说服力）

**报告必须包含以下内容**（缺一不可）：

#### 1. 测试概况
- **测试时间**：具体日期与时间
- **测试工具**：MCP 服务器名（cursor-ide-browser 或 cursor-browser-extension）
- **测试环境**：前端地址、后端地址、登录账号、选择的店铺/数据等
- **测试模块**：当前模块名称（如"模块 5：智能任务生成"）
- **对标样本**：参考的市场成品方案（如钉钉待办、飞书任务、抖音罗盘等）

#### 2. 测试步骤与结果（必须用表格或清单逐项列出）
- 按 `docs/各模块步骤总结.md` 中该模块的测试步骤，逐项列出
- 每一步必须标记 ✅ 通过 或 ❌ 失败
- 对于通过的步骤，简述验证方式（如"点击按钮后弹窗正常打开"）
- 对于失败的步骤（如果有），必须详细说明：失败现象、期望结果、实际结果

**示例表格**：

| 步骤 | 操作 | 预期 | MCP 结果 | 说明 |
|------|------|------|----------|------|
| 1 | 点击「智能生成」按钮 | 发起生成请求 | ✅ | 点击后按钮 disabled，请求已发起 |
| 2 | 验证任务标题 | 含具体数值与目标 | ✅ | 如「提升转化率至 3.5%」等 |
| 3 | 验证任务描述 | 含当前值/目标值、措施 | ✅ | 含数据来源、1-2 条措施（ZPD） |
| 4 | 验证优先级 | 紧急任务有标识 | ✅ | 有「紧急」「普通」Tab，可筛选 |

#### 3. 严苛测试清单结果（50+ 项检查项）
- 必须逐项列出「严苛测试标准」中的 7 大类检查项
- 每一项标记 ✅ 通过 或 ❌ 失败 或 ➖ 不适用
- 对于失败项（如果有），必须说明原因
- **只有全部通过**，才能判定为"无问题"

**示例清单**：

**功能完整性（核心）**
- ✅ 所有测试步骤中的操作均能成功执行
- ✅ 每个按钮/链接点击后有明确响应
- ✅ 数据提交后能正确保存并在界面上体现
- ✅ 数据加载正确，数值准确无误
- ✅ 所有 API 调用成功，无控制台报错
- ✅ 列表/表格数据渲染正确
- ✅ 表单验证规则生效

**交互体验（对标市场方案）**
- ✅ 按钮点击后有视觉反馈
- ✅ 加载时有 loading 状态
- ✅ 操作成功/失败后有 Toast 提示
- ✅ 弹窗/模态框能正确打开和关闭
- ✅ 输入框有 placeholder、聚焦样式
- ✅ 下拉菜单、选择器操作流畅
- ✅ 长列表有虚拟滚动或分页
- ✅ 响应式布局在不同屏幕尺寸下正常

**（以此类推，列出所有 7 大类）**

#### 4. 对标结论（与市场方案的具体对比）
- 必须明确列出：本模块在哪些方面**达到**市场方案水平
- 必须明确列出：本模块在哪些方面**超过**市场方案（如果有）
- 必须明确列出：本模块在哪些方面**不如**市场方案（如果有，则不应判定为"无问题"）
- 给出具体的对比数据（如加载时间、响应速度、功能数量等）

**示例**：
- **达到市场水平**：任务标题含数值、任务描述含措施、有优先级标识，与钉钉待办水平相当
- **超过市场方案**：任务描述基于 ZPD 规则生成，比通用待办更精准
- **不如市场方案**：无（所有方面均达到或超过对标方案）

#### 5. MCP 调用记录（证据）
- 必须包含至少 3 次 MCP 调用的关键 snapshot 或截图
- 必须包含 `browser_console_messages` 的调用结果（证明无控制台错误）
- 必须包含 `browser_network_requests` 的调用结果（证明 API 正常）
- 可以截取 snapshot 中的关键部分（如按钮状态、数据显示、布局结构等）

#### 6. 功能与布局总结
- 用简洁的语言总结本模块的核心功能
- 用简洁的语言总结本模块的布局结构
- 指出本模块的亮点（如果有）
- 指出本模块的待改进点（如果有，则需要说明为何判定为"无问题"）

#### 7. 实现说明
- 简要说明后端实现（如涉及的 API、数据库、业务规则）
- 简要说明前端实现（如涉及的组件、页面、状态管理）
- 如有特殊逻辑（如 ZPD 规则、异常检测等），需要说明

#### 8. 最终结论
- 明确标记：**✅ 模块 X MCP 通过，待人工终测**
- 或标记：**❌ 模块 X MCP 未通过，存在以下问题需修复：[列出问题]**
- 只有前述 1-7 项全部符合要求，才能标记为"通过"

**报告存放位置**：
- 报告可写在 `docs/` 下对应模块文档
- 或更新《各模块步骤总结》（`docs/各模块步骤总结.md`）的该模块小节
- 必须在文档中清晰标记模块状态（如 `## 📝 模块 5：智能任务生成 ✅ MCP 通过，待人工终测`）

**报告质量要求**：
- 不得出现模糊表述（如"基本正常"、"大致可用"）
- 必须用**明确的通过 ✅ / 失败 ❌** 标记
- 必须用**具体的数据和证据**支撑结论
- 报告应**详尽且有说服力**，让人工终测者能够快速理解测试结果

### 步骤 5：人工终测

- 报告输出后，流程进入**人工终测**阶段，由人工做最终确认。

## 约束（严苛执行，绝不妥协）

### 强制性约束（违反任何一项视为流程失败）

1. **步骤 2 必须实际调用 `call_mcp_tool`**：
   - 至少发起**3 次**以上 MCP 浏览器工具调用（如 browser_navigate、browser_snapshot、browser_click 等）
   - 必须调用 `browser_console_messages` 检查控制台错误
   - 必须调用 `browser_network_requests` 检查 API 请求情况
   - 不得仅写「由人工测试」或「MCP 不可用」而跳过

2. **未完成步骤 2（MCP 测试并对标）前**，不得输出「模块通过」报告：
   - 报告中必须包含**详细的测试清单**（50+ 项检查项逐项列出通过/失败）
   - 报告中必须包含**对标结论**（与市场方案的具体对比）
   - 报告中必须包含**MCP 调用记录**（至少 3 次 snapshot 或关键操作截图）

3. **发现问题时，必须先修复再重测**：
   - 不得在报告中写「通过」却留有未修复问题
   - 不得以"小问题"、"不影响主流程"、"后续优化"等理由放过问题
   - 每次修复后必须**完整重测**，不得只测"修复部分"

4. **零容忍测试标准**：
   - 任何功能缺失、交互异常、布局瑕疵、数据错误、文案问题，都视为**严重问题**
   - 对标结果不如市场方案的，即使功能正常也视为**不合格**
   - 性能不达标（首屏 > 2 秒、操作延迟 > 2 秒、列表卡顿）视为**严重问题**
   - 控制台有任何错误或警告（除已知的第三方库警告外），视为**严重问题**

5. **报告质量标准**：
   - 不得出现模糊表述（如"基本正常"、"大致可用"、"问题不大"）
   - 必须用**明确的通过 ✅ / 失败 ❌** 标记
   - 失败项必须包含：失败现象、期望结果、实际结果、证据（snapshot/截图）
   - 对标结论必须具体（如"加载速度比钉钉慢 2 秒"，而非"速度较慢"）

### 禁止行为（出现任何一项立即终止测试，要求重新执行）

- ❌ 跳过 MCP 调用，只人工测试或"目测"
- ❌ 发现问题但标记为"通过"，计划"后续优化"
- ❌ 报告中只写"测试通过"，不列出检查清单
- ❌ 对标时只说"功能齐全"，不指出与市场方案的差距
- ❌ 只测试"正常流程"，不测试边界情况和异常场景
- ❌ 发现布局、文案等"细节问题"但认为"不影响使用"而放过
- ❌ 控制台有错误但认为"不影响功能"而忽略
- ❌ 性能测试只看"能否加载出来"，不测实际加载时间
- ❌ 修复问题后只测"修复点"，不完整重测整个模块

### 测试人设要求

**作为测试执行者，必须具备以下特质**：
- 🔍 **极度较真**：1px 的偏差、0.1 秒的延迟、一个标点符号的错误，都不能放过
- 🎯 **追求完美**：不仅要求"能用"，更要求"好用"、"美观"、"专业"
- 📏 **严守标准**：对标市场成品时，不降低标准，不妥协让步
- 🧐 **怀疑一切**：不相信"应该没问题"，每个细节都要亲自验证
- 💪 **坚持原则**：发现问题绝不放行，宁可多测几轮也不降低质量

**测试时的心态**：
- 假设自己是**最挑剔的客户**，付了高价要求最好的产品
- 假设这是**上线前的最后一道关卡**，任何问题都会导致重大损失
- 假设对标方案（钉钉/飞书/抖音罗盘）的用户会来使用本系统，他们会拿来对比
- 假设每个"小问题"都会在生产环境放大 10 倍，变成"大事故"

**测试口号**：
> **「细节决定成败，完美才是及格线」**  
> **「不是能用就行，而是要做到最好」**  
> **「发现问题不可怕，放过问题才可怕」**
