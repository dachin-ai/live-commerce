# 系统层面分析与规划建议

本文档从**系统规划师**视角，对直播电商中台（lvbcsym）做整体盘点：架构与边界、系统级冲突、优化方向，以及新功能落地建议。便于后续迭代时减少冲突、保持一致性。

---

## 一、系统概览

### 1.1 技术栈与边界

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | React 18、TypeScript、Vite、Tailwind、React Query、Axios | SPA，路由与 Composer/Agent 测试友好 |
| 后端 | Node.js、Express、TypeScript、SQLite3、JWT | 单体 API，无独立队列/Worker |
| 数据 | SQLite（WAL）、文件上传（uploads）、config/script-rules/*.json | 单机部署为主 |
| 文档/流程 | docs/、.cursor/rules、.cursor/skills | 模块测试闭环、MCP 测试步骤已固化 |

### 1.2 核心数据流

- **认证**：JWT + `user_sessions` 表校验；前端 `api` 拦截器统一带 Token，401 清本地并跳转登录。
- **店铺上下文**：`StoreContext` + `selectedStoreId` 持久化到 localStorage；统计/任务/素材/导入均按 `storeId` 隔离。
- **AI 能力**：话术生成、智能待办、报告/分析/对比等集中在 `backend/src/routes/ai-refactored.ts`，依赖 `scriptLLM`、规则引擎与 DB。

### 1.3 模块进度（与各模块步骤总结一致）

- **已完成/人工终测**：模块 1～7（登录、店铺、Dashboard、布局、智能待办、分析页、执行工具）。
- **待开始**：模块 8～10（角色权限）、11（导航与路由）、12（错误处理）、13（数值范围保护）。

## 二、系统层面冲突与不一致

### 2.1 文档与代码不一致

| 项 | 文档描述 | 实际代码 | 建议 |
|----|----------|----------|------|
| AI 路由文件 | `项目结构说明.md` 写 `routes/ai.ts` | `index.ts` 挂载的是 `ai-refactored.ts` | 将文档改为 `ai-refactored.ts`，或注明「当前入口为 ai-refactored」 |
| 限流额度 | `OPTIMIZATION.md` 写「15 分钟内最多 100 个请求」 | `index.ts` 为 `rateLimitMiddleware(500, 15*60*1000)` | 将 OPTIMIZATION 改为 500，或注明「当前为 500，可按环境调低」 |
| 备份文件 | 无文档说明 | 已删除（归档前源文件已处理），项目入口以 ai-refactored.ts 为准 | ✅ 已处理 |

### 2.2 认证与权限缺口

| 位置 | 问题 | 风险 | 建议 |
|------|------|------|------|
| `version-logs.ts` POST | 注释写「仅管理员」，但未使用 `requireAdmin` | 任意登录用户可创建版本日志 | 增加 `authenticate` + `requireAdmin` |
| `config.ts` PUT `/script-rules` | 仅 `authenticate`，未 `requireAdmin` | 任意登录用户可改话术规则配置 | 若设计为仅管理员可改，则加 `requireAdmin` |
| `auth.ts` `/me`、`/me` PUT | 未使用统一 `authenticate` 中间件，手写 token 校验 | 逻辑重复、后续改鉴权易漏 | 可选：改为 `router.use(authenticate)` + 从 `req.user` 取用户 |

### 2.3 类型与契约

- **Task**：后端 `types/index.ts` 中 `description?` 可选；前端 `services/tasks.ts` 中 `description: string` 必填。若接口实际可返回空描述，前端类型应改为可选或兼容空串。
- **前后端类型分离**：Task、Store、LiveStats 等各自定义，无共享包。新字段容易只改一端导致类型与接口不一致。建议：重要实体在后端维护「契约类型」，前端通过生成或手动同步保持一致。

### 2.4 单文件过大

- `backend/src/routes/ai-refactored.ts` 体量很大（约 1700+ 行），集中了话术、待办生成、报告、分析、对比、品牌策略、流式等。可维护性与单测成本高，新人定位逻辑困难。
- 建议：按领域拆分为多个路由或子模块（如 `ai/script.ts`、`ai/tasks.ts`、`ai/report.ts`），由 `index` 或主路由聚合挂载，便于分工与回归。

## 三、优化方向

### 3.1 后端

- **限流与缓存**：当前内存限流、内存缓存；若多实例或生产部署，考虑 Redis 限流/缓存，并在文档中统一描述策略。
- **错误处理**：全局 error 中间件已存在；建议关键路由使用统一响应格式（如 `{ ok, data?, error?, code? }`），便于前端与监控统一处理。
- **日志与可观测**：关键业务（如待办生成、话术生成、数据导入）建议结构化日志（请求 id、userId、storeId、耗时），便于排查与后续接入 APM。

### 3.2 前端

- **API 超时**：`api.ts` 默认 10s；智能生成已单独 60s。其他长耗时 AI 接口（话术、报告等）若经常超时，可对对应 service 单独设 timeout，或在 api 层按 path 配置不同超时。
- **错误与 401**：401 已统一跳转登录；建议重要操作（如创建店铺、智能生成）在 catch 中给出明确 Toast，避免仅控制台报错。
- **角色与菜单**：模块 8～10 未做前，侧栏与页面已预留 `isAnchor`/`isOperator`/`isManager` 等；实现角色时与 `docs/预留接口与上线说明.md` 对齐，避免重复改 UI。

### 3.3 文档与仓库

- **文档数量多**：`docs/` 下测试报告、说明、对标文档较多，建议在 `docs/README.md` 中维护「当前必读」与「历史/归档」分类，减少新人认知负担。
- **规则与 Skill**：`.cursor/rules` 与 `module-test-workflow` Skill 已固化模块测试流程；新增模块时在「各模块步骤总结」中补一行，并注明是否需 MCP 步骤文档。

## 四、新功能实现建议

### 4.1 新增/扩展 API

- **优先在独立路由或独立文件**：避免继续把新能力全部塞进 `ai-refactored.ts`。例如：新「直播中实时助手」可单独 `routes/live-assistant.ts` 或 `ai/live-assistant.ts`，再在入口挂载。
- **认证与权限**：新接口若需登录，统一使用 `authenticate`；若仅管理员，叠加 `requireAdmin`，并在文档中写明。
- **契约先行**：新接口的请求/响应形状建议在类型或 OpenAPI 中定稿后再实现，减少前后端反复对齐。

### 4.2 新增前端页面/模块

- **店铺上下文**：需要「当前店铺」的页面继续用 `StoreProvider` 包裹，通过 `StoreContext` 取 `selectedStore`，与现有 Dashboard/分析/工具页一致。
- **角色**：实现模块 8～10 时，从 JWT 或 `/auth/me` 取 `role`，映射为 `isAnchor`/`isOperator`/`isManager`，再驱动侧栏与路由显隐，与预留接口说明一致。
- **长请求**：若新功能有长时间运行的接口（如批量导出、长时 AI），前端需 loading 与超时配置，必要时可考虑轮询或 SSE，避免单次请求过长导致超时与体验差。

### 4.3 与现有能力的衔接

- **待办与话术**：新功能若会「写任务」或「生成话术」，尽量复用现有 `POST /api/tasks`、`POST /api/ai/script` 等，并在参数上扩展（如 source、metadata），避免另起一套语义相同接口。
- **配置与特性开关**：新功能若需开关或配置，可沿用 `config` 路由或环境变量，并在 `docs/` 中集中说明（如话术 LLM、SKIP_TODO_LLM 等），避免散落。

### 4.4 测试与发布

- **模块测试流程**：新功能若属于某模块，按 `module-test-workflow` 与「各模块步骤总结」执行：后端实现 → MCP/前端验证 → 问题则回 1 → 通过则报告 → 人工终测。
- **回归**：改动 `ai-refactored.ts` 或共享中间件/DB 时，建议跑一遍模块 5（智能生成）、模块 7（话术/工具页）的自动化或 MCP 步骤，防止回退。

## 五、建议优先处理项（短期）

1. **权限**：为 `version-logs` POST 增加 `authenticate` + `requireAdmin`；确认 `config/script-rules` PUT 是否仅管理员可写，若是则加 `requireAdmin`。  
   **已处理**：version-logs POST、config PUT script-rules 均已加 `requireAdmin`。
2. **文档**：更新 `项目结构说明.md` 中 AI 路由为 `ai-refactored.ts`；更新 `OPTIMIZATION.md` 限流为 500；删除或归档 `ai.ts.backup` 并注明。  
   **已处理**：项目结构说明、OPTIMIZATION 已更新；`ai.ts.backup` 已删除。
3. **类型**：前端 `Task.description` 与后端契约对齐（可选或必填、空串处理）。
4. **规划**：若即将开发模块 8～10，先定角色与权限矩阵，再与预留的 `isAnchor`/`isOperator`/`isManager` 及侧栏/路由对应，避免实现时反复改。

## 七、小结

| 维度 | 现状 | 建议 |
|------|------|------|
| 架构 | 前后端分离、单体后端、SQLite | 保持；扩容时再考虑 Redis/多实例 |
| 冲突 | 文档与代码不一致、部分接口权限与注释不符、类型分散 | 按上文「冲突」与「优先处理项」逐项收口 |
| 可维护性 | AI 路由单文件过大、前后端类型重复 | 拆分 AI 路由；重要实体考虑共享类型或契约 |
| 新功能 | 模块 8～13 待做、预留接口已存在 | 新接口独立文件、权限统一、沿用 StoreContext 与模块测试流程 |

本文档可根据每次大版本或季度规划做一次刷新，保持「系统规划师」视角的单一事实来源。
