# 测试调试多轮循环优化

## 目标

把「测试 → 发现问题 → 调试修复 → 再测试」做成**可重复、可记录、易收敛**的多轮循环，减少重复劳动、避免漏测、便于交接。

**整体闭环**：模块级测试须按「后端 → MCP 测前端并对标 → 有问题则修后端再测 → 无问题则出报告 → 人工终测」执行，详见 **《模块测试流程-前后端闭环》**（`docs/模块测试流程-前后端闭环.md`）及 Cursor 规则 `module-test-workflow.mdc`。

---

## 多轮循环定义

```
第 N 轮：
  1. 前置检查（环境、服务、当前模块范围）
  2. 执行该模块测试步骤（按《各模块步骤总结》或对应模块清单）
  3. 记录：通过项 / 失败项 / 阻塞项
  4. 若有失败或阻塞 → 调试（修代码/配置/数据）→ 本地自验 → 进入第 N+1 轮
  5. 若全部通过 → 标记模块通过 → 记录结果 → 进入下一模块
```

**单轮耗时优化**：每轮只测**当前模块**、只修**本轮发现的问题**，避免顺手改其他模块导致回归。

---

## 优化要点

### 1. 每轮开始前（约 1 分钟）

| 检查项 | 操作 | 未通过时 |
|--------|------|----------|
| 后端 | `http://localhost:3000/health` 返回 ok | 启动 `backend`，必要时先释放 3000 端口 |
| 前端 | `http://localhost:5173` 可访问 | 启动 `frontend` |
| 当前模块 | 确认本轮只测《各模块步骤总结》中「当前模块」的步骤 | 不扩大范围 |
| 测试清单 | 打开该模块的测试清单（如 `模块345-测试执行清单.md` 或模块 N 的调试与测试文档） | 按步骤勾选，避免漏测 |

### 2. 执行测试时

- **优先按文档步骤顺序**：与《各模块步骤总结》或《完整功能测试指南》一致，便于对结果。
- **可选 MCP 自动化**：模块 3/4/5 等已用 MCP 跑过时，可复现；若遇 Cursor 序列化错误，改用**手动 + 清单勾选**。
- **失败即记**：在当轮记录里写下「步骤 X：预期 Y，实际 Z」，便于调试时定向修改。

### 3. 发现问题时（调试阶段）

- **只改与失败步骤相关的代码/配置**，改完在本机自验该步骤再进入下一轮。
- **修改点记入当轮记录**：文件 + 简要修改内容，方便后续回归和交接。

### 4. 当轮通过标准

- 该模块在《各模块步骤总结》中列出的**所有测试步骤**均为通过。
- 无新增控制台报错、无阻塞性弹窗未处理。

### 5. 进入下一模块前

- 在《各模块步骤总结》中把该模块状态更新为「✅ 已完成」。
- 在对应模块文档或 `模块345-测试执行清单.md` 等中补一句「MCP/手动 测试日期 + 结果摘要」（可选）。

## 每轮可复用检查清单（模板）

每轮开始时复制下面清单，填上模块名与轮次，执行后填结果。

```markdown
## 第 ___ 轮 | 模块 ___

- [ ] 前置：后端 3000 ok / 前端 5173 ok
- [ ] 前置：已打开本模块测试清单

### 测试结果（按步骤）

| 步骤 | 预期 | 结果（通过/失败） | 备注 |
|------|------|-------------------|------|
| 1    |      |                   |      |
| 2    |      |                   |      |
| …    |      |                   |      |

### 本轮结论

- [ ] 全部通过 → 标记模块通过，进入下一模块
- [ ] 有失败 → 记录修改点，自验后下一轮再测

### 修改点（若有）

- 文件：________ 修改：________
```

## 建议节奏

- **单轮**：约 5–15 分钟（只测当前模块、只记当前模块问题）。
- **单模块**：尽量 1–3 轮内通过；若超过 3 轮仍有同类失败，考虑拆成更小步骤或单独排查环境/数据。
- **跨模块**：当前模块通过后再切下一模块，避免并行改多模块导致回归难查。
