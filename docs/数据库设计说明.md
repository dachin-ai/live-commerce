# 数据库设计说明

本系统数据库设计对标**淘宝、字节（抖店/抖音电商）、阿里云电商方案**等业界实践，在单机 SQLite 前提下做能力与约束对齐，便于后续扩容或迁移时保持语义一致。

---

## 一、对标依据与取舍

| 维度 | 淘宝/字节等常见做法 | 本系统实现 | 说明 |
|------|---------------------|------------|------|
| 存储引擎 | 分布式/分库分表、读写分离 | SQLite 单文件 + WAL | 当前为单机部署；WAL 提升并发读与恢复 |
| 事务与锁 | 行锁、乐观锁、busy 重试 | PRAGMA busy_timeout=5000 | 降低锁等待失败率 |
| 操作可追溯 | 审计日志、操作记录 | audit_log 表 + logAudit() | 关键写操作可打点 |
| 核心表时间戳 | 创建时间 + 更新时间 | createdAt + updatedAt（users/stores/tasks） | 便于同步与排查 |
| 统计/报表 | 按店铺+日期维度、时序聚合 | stats(storeId, date) + 复合索引 | 与抖店「按日/周/月」口径一致 |
| 多租户/店铺隔离 | 按 shop_id/tenant_id 分区 | 所有业务表带 storeId 或 userId | 查询统一带 storeId 过滤 |
| 配置与键值 | 独立配置表/配置中心 | system_config(key, value) | 话术 LLM、特性开关等 |

---

## 二、表结构概览

### 2.1 核心业务表

| 表名 | 说明 | 关键字段 | 对标 |
|------|------|----------|------|
| users | 用户账号 | id, email, role, status, createdAt, updatedAt | 电商账号体系 |
| user_sessions | 会话（JWT 校验） | userId, token, expiresAt | 登录态与安全 |
| stores | 店铺/多租户主体 | id, userId, platform, region, currency, status, updatedAt | 抖店/淘宝「店铺」维度 |
| store_categories | 店铺-类目关联 | storeId, categoryId | 多对多 |
| categories | 类目树（三级） | id, parentId, level, name | 对标抖店行业类目 |
| tasks | 待办任务 | id, userId, storeId, status, priority, source, updatedAt | 运营任务与权限隔离 |
| stats | 直播统计（按日） | id, storeId, date, totalGMV, totalDuration, totalViewers, ... | 抖店罗盘口径，见《抖店直播数据项目对标》 |
| data_imports | 导入记录 | storeId, platform, fileName, recordCount, statsId | 数据溯源 |
| materials | 素材 | id, storeId, type, url | 按店铺隔离 |

### 2.2 系统与审计

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| audit_log | 操作审计 | userId, action, entityType, entityId, details, createdAt |
| system_config | 键值配置 | key, value, updatedAt |
| llm_tools | 多套 LLM 配置 | id, name, url, api_key, model |
| version_logs | 版本更新日志 | version, title, content, type |

### 2.3 认证与安全

| 表名 | 说明 |
|------|------|
| password_reset_tokens | 重置链接 token |
| password_reset_codes | 邮箱验证码 |
| user_preferences | 用户偏好（JSON） |

## 三、索引与查询模式

- **tasks**：status, priority, userId, storeId；(title, status, userId, storeId) 用于去重。
- **stats**：storeId；date；(storeId, date) 复合索引，按店铺+日期范围查询（看板、分析）。
- **stores**：status, userId。
- **audit_log**：userId；(entityType, entityId)；createdAt，便于按人/按实体/按时间排查。

---

## 四、审计日志使用

关键写操作建议调用 `logAudit()`（见 `backend/src/db.ts`），例如：

- 店铺创建/更新/删除：entityType=`store`，entityId=店铺 ID
- 用户资料/密码修改：entityType=`user`
- 配置修改（如话术 LLM）：entityType=`config`
- 数据导入：entityType=`data_import`，entityId=导入记录 ID

参数：userId, action（create/update/delete/login/export 等）, entityType, entityId, details（可选 JSON）, ipAddress（可选）。

---

## 五、统计口径与时间维度

- **stats 表**：按「店铺 + 日期」存储，一条记录表示该店铺在该自然日的汇总。与抖店「按日/周/月」筛选一致；周/月/年由应用层按 date 范围聚合。
- **指标**：totalGMV、totalDuration、totalViewers、totalOrders、averageConversionRate、gmvPerHour 等，见《抖店直播数据项目对标》与《交互定义与模块测试流程约定》。

---

## 六、扩展与迁移建议

- **枚举与字典**：当前 status、priority、platform 等为 TEXT，在《交互定义与模块测试流程约定》中约定取值；若需严格枚举可增加字典表或 CHECK 约束。
- **软删除**：stores、materials 等可通过 status='deleted' 或后续增加 deletedAt 实现软删除，便于恢复与审计。
- **迁 MySQL/PostgreSQL**：表结构与索引设计已按通用 SQL 书写，迁移时主要替换分页、时间函数与自增 ID 策略；审计与多租户隔离逻辑可复用。
