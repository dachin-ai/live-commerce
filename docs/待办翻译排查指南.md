# 待办翻译排查指南

## 问题现象

切换到英文或泰文后，待办列表仍显示中文；点击「显示为当前语言」（Show in current language）按钮后无反应。

---

## 排查步骤

### 1. 打开浏览器控制台（F12）

按 **F12** 打开开发者工具，切换到 **Console** 标签页。

### 2. 刷新页面并观察日志

刷新页面（Ctrl+R），查看控制台是否有以下日志：

```
[翻译待办 useEffect] 检查条件 {needsTranslate: true, hasStore: true, currentLocale: "en-US", ...}
```

**如果看到**：
- `needsTranslate=false，跳过` → 说明系统认为不需要翻译（可能已有缓存或判断错误）
- `无店铺，跳过` → 需要先选择一个店铺
- `中文，跳过` → 当前语言是中文，不需要翻译
- `已为此 locale 自动翻译过，跳过` → 已尝试过自动翻译（可能失败了）

**如果看到**：
```
[翻译待办 useEffect] ✅ 触发自动翻译 for locale: en-US
[翻译待办] 开始翻译 {storeId: "xxx", locale: "en-US"}
[翻译待办] API 返回 {translated: 9, total: 9}
```
→ 说明翻译成功！如果待办仍显示中文，可能是缓存或显示逻辑问题。

**如果看到错误**：
```
翻译待办失败 Error: Network Error
```
→ 网络问题或后端未响应。

```
API 返回 {translated: 0, total: 9, error: "QUOTA_EXCEEDED"}
```
→ 触发每日限额，会有 Toast 提示。

### 3. 手动点击「显示为当前语言」按钮

如果自动翻译未触发，点击待办区域标题栏右侧的翻译按钮（图标是循环箭头），观察：

- **按钮变为"翻译中…"**：正在调用 API
- **几秒后弹出 Toast**：显示翻译结果或错误信息
- **待办列表刷新**：翻译成功的任务会显示英文/泰文

### 4. 查看后端日志

如果前端日志显示调用了 API 但返回异常，切换到后端控制台查看：

```
[翻译待办] userId=user-1, storeId=xxx, locale=en-US
[翻译待办] 查询到 9 条待办
[翻译待办] 已翻译 1/9: 提升周播蓝筹 -> Improve Weekly Blue Chips
[翻译待办] 已翻译 2/9: ...
[翻译待办] 完成：translated=9, total=9, quotaExceeded=false
```

**常见错误**：
```
Error: Cannot find module 'google-translate-api-x'
```
→ 包未安装，运行：`cd backend && npm install`

```
[translate] Error: 429 Too Many Requests
```
→ 触发限额（Google Translate 或 MyMemory）。

## 常见问题

### Q1: 自动翻译未触发

**可能原因**：
1. **needsTranslate 为 false**  
   - 检查：所有待办的 `title_i18n[currentLocale]` 已存在 → 说明已翻译过，但前端显示逻辑可能有问题。
   - 解决：查看 `getTaskDisplayTitle(task, currentLocale)` 是否正确读取 `title_i18n`。

2. **lastAutoTranslateLocaleRef 已记录当前 locale**  
   - 如果之前尝试过翻译（成功或失败），ref 会记住该 locale，不会再自动触发。
   - 解决：点击「显示为当前语言」按钮手动触发，或刷新页面重置 ref。

3. **useEffect 依赖问题**  
   - 如果 `handleTranslateToCurrentLanguage` 变更频繁，可能导致 effect 不触发或频繁触发。
   - 当前已用 `useCallback` 稳定该函数。

### Q2: 点击翻译按钮无反应

**可能原因**：
1. **按钮被禁用（translating=true）**  
   - 如果上次翻译未结束，按钮会禁用。
   - 解决：刷新页面重置状态。

2. **API 请求失败但未显示错误**  
   - 查看浏览器控制台 Network 标签，搜索 `/api/tasks/translate-for-locale`，查看响应状态。
   - 如果是 401：token 过期，需要重新登录。
   - 如果是 500：后端错误，查看后端日志。

3. **后端翻译接口异常**  
   - 后端日志无 `[翻译待办]` 开头的输出 → 请求未到达后端。
   - 检查网络或 API 路由是否正确注册。

### Q3: 翻译后仍显示中文

**可能原因**：
1. **title_i18n 未写入数据库**  
   - 后端日志显示 `translated=0` → 可能所有任务的 `title_i18n[locale]` 已存在（跳过了）。
   - 或 `UPDATE tasks SET title_i18n = ...` 语句失败。

2. **前端未正确读取 title_i18n**  
   - `parseTaskI18n` 可能解析失败（如果数据库返回的不是 JSON 字符串）。
   - 检查：`console.log(tasks[0].title_i18n)` 查看原始值。

3. **locale key 不匹配**  
   - 前端用 `'en-US'`，后端存的是 `'en'` → 读不到。
   - 当前实现：前端传 `'en-US'`，后端存 `title_i18n['en-US']`，显示时也用 `'en-US'` 取。应该一致。

## 手动测试

### 测试翻译接口

在浏览器控制台执行（需先登录获取 token）：

```js
const token = localStorage.getItem('token');
fetch('/api/translate', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({
    text: '提升周播蓝筹',
    targetLang: 'en-US',
    sourceLang: 'zh-CN'
  })
}).then(r => r.json()).then(d => console.log('翻译结果:', d));
```

**预期输出**：
```json
{ "translatedText": "Improve Weekly Blue Chips" }
```

### 测试批量翻译待办

```js
const token = localStorage.getItem('token');
const storeId = localStorage.getItem('selectedStoreId');
fetch('/api/tasks/translate-for-locale', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({
    storeId,
    locale: 'en-US'
  })
}).then(r => r.json()).then(d => console.log('批量翻译结果:', d));
```

**预期输出**：
```json
{ "translated": 9, "total": 9 }
```

或触发限额时：
```json
{ "translated": 3, "total": 9, "error": "QUOTA_EXCEEDED" }
```

## 降级方案（限额用完时）

### 临时方案

1. **切换回中文界面**：待办以原文（中文）显示，功能不受影响。
2. **明日再试**：Google Translate 非官方接口的限制通常是按日重置。

### 长期方案

如果频繁触发限额，可考虑：

#### 方案 A：切换为官方 Google Cloud Translation API

- **优点**：稳定、无限额风险（按量付费，每月有一定免费额度）。
- **步骤**：
  1. 注册 Google Cloud，启用 Translation API，获取 API key。
  2. `cd backend && npm install @google-cloud/translate`
  3. 在 `.env` 中添加 `GOOGLE_TRANSLATE_API_KEY=your_key`
  4. 修改 `backend/src/utils/translate.ts`：
     ```ts
     import { Translate } from '@google-cloud/translate/v2'
     const client = new Translate({ key: process.env.GOOGLE_TRANSLATE_API_KEY })
     
     export async function translateText(text, targetLang, sourceLang) {
       // ... cache check ...
       const [translation] = await client.translate(text, {
         from: toGoogleLang(sourceLang),
         to: toGoogleLang(targetLang),
       })
       // ... cache save ...
       return translation
     }
     ```

#### 方案 B：部署 LibreTranslate（自托管，无限额）

- **优点**：开源、自托管、完全免费、无限额。
- **步骤**：
  1. Docker 部署 LibreTranslate：
     ```bash
     docker run -d -p 5000:5000 libretranslate/libretranslate
     ```
  2. 修改 `backend/src/utils/translate.ts`：
     ```ts
     const url = `http://localhost:5000/translate`
     const resp = await fetch(url, {
       method: 'POST',
       headers: { 'Content-Type': 'application/json' },
       body: JSON.stringify({
         q: text,
         source: toLibreTranslateLang(sourceLang),
         target: toLibreTranslateLang(targetLang)
       })
     })
     const data = await resp.json()
     return data.translatedText
     ```

#### 方案 C：增加缓存到数据库（持久化）

- **当前**：翻译结果缓存在 `tasks.title_i18n` + 内存 Map（重启后 Map 清空）。
- **改进**：在 `backend` 新建 `translation_cache` 表：
  ```sql
  CREATE TABLE translation_cache (
    text_hash TEXT,
    source_lang TEXT,
    target_lang TEXT,
    translated_text TEXT,
    created_at TEXT,
    PRIMARY KEY (text_hash, source_lang, target_lang)
  )
  ```
- 翻译前先查该表，有则用缓存；无则调用 API 并存入表。
- **效果**：重启后仍有缓存，跨请求共享，进一步减少外部 API 调用。
