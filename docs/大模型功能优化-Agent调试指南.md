# 大模型功能优化 Agent 调试指南

本指南覆盖两类「大模型功能优化」相关对象：

1. **Cursor 侧栏中的「大模型功能优化」Agent**（本仓库内的 AI 会话/代理）  
   - 行为规范已写入 **`.cursor/rules/大模型功能优化.mdc`**。使用该 Agent 时，请在对话中 **@ 该规则**（如 `@.cursor/rules/大模型功能优化.mdc`）或确保 Cursor 已为该 Agent 关联该规则，以便其遵守职责边界、专业输出约束与文档同步要求。  
   - 规则会在编辑 `scriptLLM`、`ai-refactored`、`ai/script` 及 Coze/待办相关文档时自动生效（globs 匹配）。

2. **待办智能生成所用的 Coze 智能体**（在 Coze 控制台可能命名为「大模型功能优化」或「待办助手」）  
   - 若该 Bot 表现不专业（输出分析段落、mermaid、错误平台、泛泛标题等），按下列项逐条排查与修正。

---

## 一、Agent 定义与范围

| 项 | 说明 |
|----|------|
| **名称** | 大模型功能优化 Agent / 待办生成 Coze Bot |
| **职责** | 仅根据【用户请求】中的店铺与数据，生成**待办任务 JSON**，不做分析、不写报告、不画图。 |
| **输出** | 有且仅有一个 JSON 对象：`{"tasks":[{"title":"","description":"","priority":"urgent或normal"}]}`，**直接以 `{"tasks":[` 开头**，无任何前缀、后缀、markdown、mermaid。 |
| **调用方** | 后端 `generateIntelligentTodosWithLLM` → `scriptLLM.callLLMOnce` → Coze `stream_run`。 |

---

## 二、常见「不专业」表现与修复

| 表现 | 原因 | 修复方式 |
|------|------|----------|
| 输出「### 分析」「建议如下」等段落 | Bot 当成了「分析型」任务 | 在 Coze **人设/系统提示**最前面加：**「本 Bot 仅输出待办 JSON，禁止输出分析、建议、总结或任何非 JSON 内容。」**；后端已强调「仅输出一个 JSON」，若仍出现可把后端 systemPrompt 首句改为更强硬（见下文「专业化系统提示」）。 |
| 输出 mermaid 图、流程图 | 模型倾向「可视化」 | 在人设中明确写：**「严禁输出 mermaid、流程图、代码块或 markdown。」** 后端已有类似约束，Coze 侧务必同步。 |
| 输出 \`\`\`json ... \`\`\` | 带了 markdown 代码块 | 强调：**「直接输出裸 JSON，不要用 \`\`\` 包裹。」** |
| 任务描述里出现「抖音」而店铺是 TikTok | 平台与数据来源不一致 | 在人设中写：**「任务描述中的数据来源必须与【用户请求】里【店铺】后的平台一致：TikTok 店铺只写 TikTok，抖音店铺只写抖音，快手只写快手，不得混用。」** 后端 systemPrompt 已有此条，Coze 侧需一致。 |
| 标题泛泛（如「提升转化率至 3.5%」「增加直播时长」） | 未按「场景化、多维度」要求 | 在人设中写：**「标题必须场景化具体，如『今晚7点加播2小时』『每10分钟福袋拉互动』，禁止泛泛的『提升X至Y%』『增加时长至X小时』类模板句。」** 后端已有，Coze 需同步。 |
| 多条任务标题/描述雷同 | 未按「不同维度」区分 | 强调：**「每条任务对应不同维度（时段/加播、互动/福袋、选品/上新、引流、节日备货、时长、转化、客单价、观看人数等），标题和描述不可重复或雷同。」** |
| 回复过慢或超时 | 生成长文本或思考过长 | 后端已设 40s 超时；Coze 侧可限制**最大回复长度**（如 2000 字），并再次强调「只输出 JSON，不要展开」。 |

## 三、专业化系统提示词（可整段贴到 Coze 人设）

在 Coze 该 Bot 的**人设 / 系统提示**中，使用下面整段（与后端逻辑对齐，且更强调「只出 JSON、不分析」）：

```
你是专业的直播电商待办助手。本 Bot 仅输出待办任务 JSON，禁止输出任何分析、建议、总结、mermaid 图、markdown 代码块或其它非 JSON 内容。

【输出格式】
有且仅有一个 JSON 对象，直接以 {"tasks":[ 开头，无任何前缀或后缀。格式：{"tasks":[{"title":"","description":"","priority":"urgent或normal"}]}，条数按需，无上限。

【硬性约束】
1. 数据来源与【用户请求】中【店铺】后的平台一致：TikTok 店铺只写 TikTok，抖音只写抖音，快手只写快手，不得混用。
2. 每条任务针对不同维度（时段/加播、互动/福袋、选品/上新、引流/短视频、节日备货、时长、转化、客单价、观看人数等），标题和描述不可重复或雷同。
3. 标题≤30字，场景化具体（如「今晚7点加播2小时」「每10分钟福袋拉互动」），禁止泛泛的「提升X至Y%」「增加时长至X小时」类模板句。
4. 描述≤120字，含 1～2 个具体动作与可量化预期。
5. 待办须结合该店铺的类目、目标人群、品牌定位与价格区间，避免行业通用空话。
6. 【已有待办】中已列出的标题，不得重复或雷同生成。
```

## 四、自检清单（调试时逐项打勾）

- [ ] **Coze 人设**：已粘贴或合并「专业化系统提示词」；首句明确「仅输出 JSON，禁止分析/mermaid/markdown」。
- [ ] **平台一致**：人设中已写「数据来源与店铺平台一致」，并用 `docs/Coze待办生成-测试消息.md` 中**抖音**与**TikTok**两种示例各测一次，确认描述中无混用平台。
- [ ] **回复格式**：用测试消息发送后，回复**直接以 `{"tasks":[` 开头**，无 \`\`\`json、无 mermaid、无「分析」「建议」等段落。
- [ ] **标题与维度**：生成的标题场景化、多条对应不同维度；无「提升X至Y%」等泛泛句。
- [ ] **后端日志**：若线上仍不专业，查看 `backend/generate-tasks.log` 的「发出 message 前300字」与 `backend/coze-debug.log`，确认请求发往的 URL 与测试时一致、message 以「你是…待办助手」或「仅输出…JSON」开头。
- [ ] **多轮记忆**：Coze 若开启多轮记忆，建议对该 Bot 关闭或确保首轮回复不依赖历史，避免历史对话干扰「只输出 JSON」。

## 五、后端已做的约束（供对照）

后端 `ai-refactored.ts` 中 `generateIntelligentTodosWithLLM` 的 systemPrompt 已包含：

- 仅输出一个 JSON，严禁 markdown/mermaid/分析段落。
- 数据来源与店铺平台一致（TikTok/抖音/快手）。
- 标题场景化、多维度、≤30 字；描述≤120 字、含具体动作与可量化预期。
- 结合店铺类目、目标人群、品牌定位、价格区间；与【已有待办】不重复。

若 Coze 侧人设与上述不一致，**以本调试指南第三节「专业化系统提示词」为准**，在 Coze 中覆盖或合并，并与后端保持一致。

## 六、测试步骤

1. 打开 `docs/Coze待办生成-测试消息.md`，复制**方式一**的完整测试消息（含【店铺】为抖音或 TikTok 的示例）。
2. 在 Coze 该 Bot 的「预览/对话」中粘贴为**一条用户消息**发送。
3. 检查回复是否：① 仅一个 JSON、直接以 `{"tasks":[` 开头；② 无代码块、无 mermaid、无分析段落；③ 描述中数据来源与【店铺】平台一致。
4. 若通过，再在应用内点击「智能生成」，观察是否仍出现不专业输出；若出现，查 `generate-tasks.log` 与 `coze-debug.log` 核对请求内容与 URL。
