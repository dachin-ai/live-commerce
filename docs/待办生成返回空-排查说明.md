# 待办生成返回空 - 排查说明

当出现「第 2 次重试调用 LLM（前次返回空）」或「Coze/LLM 2 次调用均返回空」时，表示**流式请求已发出，但从 Coze/Agent 的 SSE 里没有拿到任何正文**（聚合后为空）。按下面顺序排查。

---

## 0. 0 字节（空 body）时先做这几步

若日志出现 **`[empty_stream_no_payloads]`** 且 **`totalBytesReceived: 0`**（Coze 返回 200 但 body 为空），多半是**鉴权或 URL** 问题：

1. **看同一条日志里的 `hasAuth`**  
   - 若为 `false`：请求未带 API Key。请任选其一：  
     - 在 **管理员 → LLM 配置** 中填写 **API 密钥** 并保存；  
     - 或在 **`backend/.env`** 中配置 **`AGENT_API_KEY`**（*.coze.site 会作为兜底）、或 **`SCRIPT_LLM_API_KEY`**，然后**重启后端**。
   - 若为 `true` 仍 0 字节：可能是 Key 错误或已失效、或 Coze 发布站点对该 Key 有调用限制（如仅创建者可用）。**若当前登录为非管理员账号**：请管理员确认「LLM 配置」中已为该账号开通使用权限（允许用户列表）；并用同一 API Key 在 Postman 直接请求该 URL 的 `POST /stream_run`，验证是否返回正文（本系统与管理员共用同一套 URL/Key，不按账号区分）。

2. **确认 URL**  
   - 日志中的 `urlHint` 应为 Coze 的 **stream_run** 地址（本系统会自动在末尾加 `/stream_run`）。  
   - 若为 *.coze.site，**不要**在 `.env` 中设置 `AGENT_API_BODY=1`（coze.site 使用 legacy 体，无需该变量）。

3. **确认后端已加载 .env**  
   - 若用环境变量，修改 `.env` 后必须**重启后端**；且需保证 `backend/.env` 在启动时被加载（见 `backend/src/loadEnv.ts`）。

4. **Coze 侧**  
   - 发布站点/Agent 是否要求鉴权、当前 Key 是否有该 Bot 的访问权限。

---

## 1. 为什么是「返回空」

- 本系统调用 Coze 的 **stream_run**，按 SSE 事件**逐块**收取 `answer` / `content` 等字段并**拼接成一段完整文本**，再解析 JSON。
- 若 Coze 只返回了 `message_start`、`message_end`，且其中的 **`content.answer`、`content.thinking` 均为 null**，则没有任何可拼接的正文 → 我们视为「返回空」。
- 因此「返回空」= **对方没有在流式事件里输出正文**，不是请求没发出去。

---

## 2. 常见原因与对应处理

| 原因 | 说明 | 建议 |
|------|------|------|
| **Bot 未在 answer 中输出** | Coze Bot 只发了 start/end，没有发 `type: "answer"` 且带 `answer` 或 `content.answer` 的流式块 | 在 Coze 控制台检查该 Bot：人设/工作流必须配置为**在 answer 中流式输出**待办 JSON，本系统仅从 answer/delta 拼接正文 |
| **请求体格式不一致** | 《Coze 对接协议》规定入参为 **content.query.prompt[0].content.text**（legacy 体）。本系统对 **\*.coze.site** 默认已用该格式。若接入的是**自建 Agent 且仅支持** `{ content, session_id }`，再设 **`AGENT_API_BODY=1`** | 看 coze-debug.log 的 **bodyKind**：legacy=符合协议；agent=仅当对方明确要求时使用 |
| **SSE 事件/字段名不同** | Agent 用别的 event 或字段名输出正文（如 `content.reply`、`content.output`） | 查看 `backend/coze-debug.log` 中本次请求的 **empty_stream_payloads**、**messageEndContentKeys**，把实际字段名反馈给开发，以便在解析中增加兼容 |
| **超时** | 流式在约定时间内未收到足够内容即中止 | 看 coze-debug.log 中该次请求的 `stream_done` 是否 **aborted: true**。若同时出现 `[stream_timeout]`，可看到 receivedTypes。待办流式超时默认 **5 分钟**；可在 `backend/.env` 中设置 **`COZE_TODO_TIMEOUT_MS=600000`**（10 分钟）等更大值后重启后端。 |
| **API 密钥/权限** | Token 无效或该 Bot 未对当前环境开放 | 在「用户管理 → LLM 配置」中确认 API 密钥正确；在 Coze 侧确认发布/权限设置 |

## 3. 建议先做的两步

### 3.1 请求体格式说明

- **\*.coze.site（含电商数据分析专家 Agent）**：本系统默认发送 **legacy 体**（`content.query.prompt[0].content.text`），与《Coze 对接协议 - 最终落地方案》一致，**无需**设置 `AGENT_API_BODY=1`。若 URL 为 \*.coze.site 却仍返回空或异常，请检查 **`backend/.env`** 是否误设 **`AGENT_API_BODY=1`**（coze.site 应使用 legacy，无需且不应设该变量）。
- **空 body（totalBytesReceived=0）**：多为鉴权问题。请在「管理员 → LLM 配置」中填写 API Key，或于 `backend/.env` 中配置 **`AGENT_API_KEY`**（coze.site 会作为兜底使用）。
- 仅当接入的 **自建 Agent** 明确只支持 `{ content, session_id }` 时，再设 **`AGENT_API_BODY=1`**。

### 3.2 看 coze-debug.log 确认流里有什么

1. 打开 **`backend/coze-debug.log`**
2. 找到**最近一次**待办生成对应的两条记录：
   - `[request]`：本次请求（messageLen、url、taskType: todo）
   - 紧接着的 `[stream_done]`：`yieldedChunks`、`yieldedLen`（若为 0 即为本次「返回空」）
3. 若本次是空，再往上看是否有 **`[empty_stream_payloads]`**：
   - 其中 **sampleTypes** 会列出收到的 SSE 事件类型（如 message_start、message_end）
   - **messageEndContentKeys** 会列出 `message_end` 里 `content` 的键名（若只有 answer、thinking 且均为 null，说明 Bot 没有在 content 里填正文）

根据这些可以判断是「Bot 没输出」还是「输出在了我们未解析的字段里」。
