# 待办生成返回空 - 排查说明

当出现「第 2 次重试调用 LLM（前次返回空）」或「Coze/LLM 2 次调用均返回空」时，表示**流式请求已发出，但从 Coze/Agent 的 SSE 里没有拿到任何正文**（聚合后为空）。按下面顺序排查。

---

## 1. 为什么是「返回空」

- 本系统调用 Coze 的 **stream_run**，按 SSE 事件**逐块**收取 `answer` / `content` 等字段并**拼接成一段完整文本**，再解析 JSON。
- 若 Coze 只返回了 `message_start`、`message_end`，且其中的 **`content.answer`、`content.thinking` 均为 null**，则没有任何可拼接的正文 → 我们视为「返回空」。
- 因此「返回空」= **对方没有在流式事件里输出正文**，不是请求没发出去。

---

## 2. 常见原因与对应处理

| 原因 | 说明 | 建议 |
|------|------|------|
| **Bot 未在 answer 中输出** | Coze Bot 只发了 start/end，没有发 `type: "answer"` 且带 `answer` 或 `content.answer` 的流式块 | 在 Coze 控制台检查该 Bot：人设/工作流是否要求「在 answer 中流式输出」；若只配置了工具调用，需确保工具结果被写入 Bot 的回复（answer） |
| **请求体格式不一致** | 《Coze 对接协议》规定入参为 **content.query.prompt[0].content.text**（legacy 体）。本系统对 **\*.coze.site** 默认已用该格式。若接入的是**自建 Agent 且仅支持** `{ content, session_id }`，再设 **`AGENT_API_BODY=1`** | 看 coze-debug.log 的 **bodyKind**：legacy=符合协议；agent=仅当对方明确要求时使用 |
| **SSE 事件/字段名不同** | Agent 用别的 event 或字段名输出正文（如 `content.reply`、`content.output`） | 查看 `backend/coze-debug.log` 中本次请求的 **empty_stream_payloads**、**messageEndContentKeys**，把实际字段名反馈给开发，以便在解析中增加兼容 |
| **Bot 走 tool_request** | Coze 先发 `tool_request`（调工具），正文在后续 **tool_response** 里；流在收到 tool_response 前被超时中断 | 看 coze-debug.log 的 **empty_stream_payloads.sampleTypes** 是否含 `tool_request`。待办场景已改为 **180s** 流式超时；若仍超时，在 Coze 侧缩短工具耗时或再增大 `timeoutMs` |
| **只有 tool_request、没有 tool_response** | 流里只出现 message_start + tool_request，**从未出现 tool_response**；本系统在 180s 后超时断开（aborted:true） | **优先建议**：与《Coze对接说明》表格一致，让 Coze 待办改为**在 answer/delta 中直接流式输出 JSON**，不依赖 tool。若必须用工具，再查 Coze 侧：工具是否被调用、是否报错、是否在同一条 stream 里回写 tool_response。详见下文「4. 只有 tool_request 无 tool_response」 |
| **超时** | 流式在约定时间内未收到足够内容即中止 | 看 coze-debug.log 中该次请求的 `stream_done` 是否 **aborted: true**。若同时出现 `[stream_timeout]`，可看到 receivedTypes、hadToolRequest、hadToolResponse，便于区分「超时前已收到 tool_response」与「一直没收到 tool_response」 |
| **API 密钥/权限** | Token 无效或该 Bot 未对当前环境开放 | 在「用户管理 → LLM 配置」中确认 API 密钥正确；在 Coze 侧确认发布/权限设置 |

---

## 3. 建议先做的两步

### 3.1 请求体格式说明

- **\*.coze.site（含电商数据分析专家 Agent）**：本系统默认发送 **legacy 体**（`content.query.prompt[0].content.text`），与《Coze 对接协议 - 最终落地方案》一致，**无需**设置 `AGENT_API_BODY=1`。若 URL 为 \*.coze.site 却仍返回空或异常，请检查 **`backend/.env`** 是否误设 **`AGENT_API_BODY=1`**（coze.site 应使用 legacy，无需且不应设该变量）。
- **空 body（totalBytesReceived=0）**：多为鉴权问题。请在「管理员 → LLM 配置」中填写 API Key，或于 `backend/.env` 中配置 **`AGENT_API_KEY`**（coze.site 会作为兜底使用）。
- 仅当接入的 **自建 Agent** 明确只支持 `{ content, session_id }` 时，再设 **`AGENT_API_BODY=1`**。

### 3.2 看 coze-debug.log 确认流里有什么

1. 打开 **`backend/coze-debug.log`**
2. 找到**最近一次**待办生成对应的两条记录：
   - `[request]`：本次请求（messageLen、url、taskType: todo）
   - 紧接着的 `[stream_done]`：`yieldedChunks`、`yieldedLen`（若为 0 即为本次「返回空」）
3. 若本次是空，再往上看是否有 **`[empty_stream_payloads]`**：
   - 其中 **sampleTypes** 会列出收到的 SSE 事件类型（如 message_start、message_end）
   - **messageEndContentKeys** 会列出 `message_end` 里 `content` 的键名（若只有 answer、thinking 且均为 null，说明 Bot 没有在 content 里填正文）

根据这些可以判断是「Bot 没输出」还是「输出在了我们未解析的字段里」。

---

## 4. 只有 tool_request、无 tool_response（等待足够久仍无待办）

若 **sampleTypes** 仅有 `message_start`、`tool_request`，且 **stream_done** 为 `aborted: true`，说明本系统在等待 **tool_response** 时触发了流式超时（默认 180s）并断开。若你确认等待时间已足够（例如已等满 3 分钟）且不想再延长超时，则问题通常是：**Coze 从未在同一条 SSE 流里下发 tool_response**。

**优先建议（与《Coze对接说明-待办与话术入参出参》一致）**：本系统**未**要求 Coze 调用工具；走 tool 是 Bot 工作流配置所致。请让 Coze 侧将待办场景改为**在 `type: "answer"` 或 `message.delta` 中直接流式输出 JSON**（`{"tasks":[...]}`），不经过 tool_request→tool_response，即可避免「只收到 tool_request、永远收不到正文」的问题。

若必须保留工具调用，再按下面在 **Coze 侧** 排查：

1. **工具是否被调用**  
   在 Coze  Bot 运行/调试日志中确认：收到本次用户消息后，是否触发了 `generate_todo_tasks`（或你配置的待办工具）的调用。若没有触发，检查 Bot 工作流/节点是否在「用户消息」后正确调用了该工具。

2. **工具是否执行成功**  
   若工具被调用，查看该次调用的结果：是否因参数错误（如 **store_data 不是合法 JSON**）而失败。失败时 Coze 可能不会在流里推送 `tool_response`，或只推送错误信息。确保 Bot 从用户消息里按《Coze待办生成-调试说明》正确提取【store_data】并传入工具。

3. **发布站点/API 是否在同流返回 tool_response**  
   部分 Coze 发布方式或自建网关可能不会在 **同一 stream_run 流**里把工具执行结果以 `type: "tool_response"` 事件写回。若你的接入方式要求「轮询结果」或「另接口取结果」，本系统当前只消费同一条流，无法拿到 tool_response，需 Coze/发布侧改为在同一条 SSE 流中返回 tool_response。

本系统在流式超时触发时会打 **`[stream_timeout]`** 日志（含 `receivedTypes`、`hadToolRequest`、`hadToolResponse`），便于确认「已收到 tool_request 但从未收到 tool_response」。
