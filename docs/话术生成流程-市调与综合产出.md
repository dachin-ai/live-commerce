# 话术生成流程：先市调再交互 LLM，最后综合产出

**目标**：避免一次性话术，从用户角度出发，结合用户提供的信息与店铺数据，形成实测有效的话术逻辑和规则；注意每个平台的违规点；先市调再交互 LLM，最后综合产生话术。

---

## 一、整体流程

```
用户输入（产品名、价格、卖点、人群、话术类型、语言）
        +
店铺数据（店铺名、平台、品类、区域、近期统计）
        ↓
   【1. 市调】整合用户输入 + 店铺数据 + 平台合规 + 品类最佳实践
        ↓
   【2. 交互 LLM】（可选）根据市调摘要生成话术初稿
        ↓
   【3. 综合产出】模板/LLM 初稿 + 平台合规检查 + 数据依据拼接
        ↓
   最终话术（含合规提示、数据来源说明）
```

- **市调**：规则层完成，不依赖 LLM；产出结构化「市调上下文」与自然语言「市调摘要」。
- **交互 LLM**：可选；接入后使用市调摘要作为 LLM 输入，产出初稿；当前未接入时使用规则模板产出初稿。
- **综合产出**：在规则层完成；对初稿做平台合规检查、拼接店铺与数据依据、附加合规提示。

---

## 二、市调（第一步）

**位置**：`backend/src/rules/scriptResearch.ts`  
**入口**：`runScriptResearch(userInput, storeContext)`

### 2.1 输入

- **userInput**：用户提交的产品名、价格、卖点、目标人群、话术类型、语言等。
- **storeContext**：由路由层从 DB 拉取的店铺信息（店铺名、平台、区域、品类、目标人群、价格带、近期统计等）。

### 2.2 市调内容

1. **目标人群解析**（避免直接使用用户输入的原文）  
   用户输入的「目标人群」（如「资深中产、多猫家庭」）会先做解析，产出话术可用的：
   - **称呼**（addressTerm）：如「家里好几只主子的铲屎官们」「注重品质的家人们」，不直接使用「资深中产、多猫家庭」进话术正文。
   - **痛点/需求短语**（painPointsHint）：如「多只猫的卫生、空间不好管理，笼子要耐用又好清洁」「既要品质靠谱又要省心耐用」，用于「有没有正在为XXX发愁」「现实不满需求」等表述。  
   规则见 `backend/src/rules/scriptAudienceAnalysis.ts`，支持多猫家庭、资深中产、铲屎官、宝妈、女性、学生、上班族等关键词，并可结合产品名（如宠物用品）做强化。

2. **平台合规**  
   根据店铺的「平台」与「品类」解析平台规则（`platformCompliance.ts`）：
   - 禁用词、慎用词
   - 违规类型
   - 话术风格建议
   - 品类额外约束（如美妆慎用「美白」「祛斑」等）

3. **品类最佳实践**  
   根据店铺品类匹配「品类话术最佳实践」：
   - 话术结构建议
   - 禁忌与替代说法
   - 从用户角度、可复用、实测有效的写法

3. **数据依据**  
   从店铺近期统计生成「数据参考」摘要（GMV、观看、转化率等），供话术标明来源与节奏参考。

5. **市调摘要（供 LLM）**  
   将上述内容整理成自然语言「市调摘要」，供后续 LLM 一次性阅读与生成。

### 2.3 输出

- **ScriptResearchContext**：包含 userInput、storeContext、platformRule、categoryPractices、dataHint、summaryForLLM。
- **buildLLMSystemPrompt(research)** / **buildLLMUserMessage(research)**：接入 LLM 时，用市调结果构建系统提示与用户消息。

## 三、平台合规与违规点

**位置**：`backend/src/rules/platformCompliance.ts`

### 3.1 支持平台

| 平台 ID   | 平台名称           |
|-----------|--------------------|
| douyin    | 抖音/抖音电商      |
| taobao    | 淘宝/淘宝直播      |
| kuaishou  | 快手/快手电商      |
| tiktok    | TikTok / TikTok Shop |
| wechat    | 微信视频号/微信直播 |
| default   | 通用（兜底）        |

### 3.2 规则内容（每个平台）

- **bannedWords**：禁用词，一经使用易触发限流/违规。
- **cautionWords**：慎用词，需有依据或人工复核。
- **violationTypes**：平台特别关注的违规类型说明。
- **styleTips**：建议话术风格（用户视角、可复用、避免一次性夸张）。
- **categoryExtra**：按品类追加的禁用/慎用（如美妆、保健食品、母婴等）。

### 3.3 合规检查

- **getPlatformCompliance(platformName, category)**：解析平台 + 品类，返回合并后的规则。
- **checkPlatformCompliance(content, platformName, category)**：对产出话术做一次扫描，返回是否通过、命中禁用/慎用词、建议修改说明。

合规结果用于「综合产出」：不通过或存在慎用词时，在文末附加合规提示，不阻断产出，提醒人工复核。

---

## 四、综合产出（第二步 + 第三步）

**位置**：`backend/src/rules/scriptSynthesis.ts`  
**入口**：`synthesizeScript(research)`

### 4.1 步骤

1. **生成初稿**  
   当前：使用规则模板 `buildScriptContent(scriptType, language, params)` 生成话术初稿。  
   接入 LLM 后：可在此改为「根据 research.summaryForLLM 调用 LLM，得到初稿」。

2. **平台合规检查**  
   使用 `checkPlatformCompliance(content, platform, category)` 对初稿做扫描。

3. **拼接店铺与数据依据**  
   若有店铺：在话术前拼接「【基于店铺】店铺名（平台）（品类）」及市调中的「【数据参考】…」。

4. **合规提示**  
   若存在禁用词或慎用词：在文末追加「合规提示」与平台建议，不删除原文，只提醒人工复核。

### 4.2 输出

- **content**：最终话术正文。
- **compliancePass**：是否通过平台合规（无禁用词）。
- **complianceWarnings**：合规警告与建议。
- **dataSourceNote**：数据来源说明（便于前端或运营标注）。

## 五、可复用话术逻辑（避免一次性话术）

### 5.1 规则驱动

- 话术**结构**由话术类型与品类实践固定（如圈人群-塑品-打消顾虑-利益点-售后-逼单）。
- 话术**内容**由市调结果填充：用户输入 + 店铺数据 + 平台合规 + 品类实践。
- 同一话术类型、同一品类可多次生成，仅替换产品与数据，**逻辑可复用、可迭代**。

### 5.2 从用户角度出发

- 需求激发区分「美好生活」与「现实不满」。
- 卖点提炼区分高意向（核心卖点）与中低意向（次要卖点）。
- 打消顾虑、利益点、售后、逼单均按用户决策路径设计，而非单次噱头话术。

### 5.3 实测有效

- 品类最佳实践来源于市调与行业共识（见 `scriptResearch.ts` 中 CATEGORY_PRACTICES）。
- 平台合规来源于各平台规则与违规案例，持续可配置、可迭代。
- 数据依据来自店铺真实统计，话术可标明「近期转化率约 X%」等，增强可信度。

### 5.4 非一次性

- 每次生成都走「市调 → 综合产出」，用**同一套规则**与**当次用户输入、店铺数据**生成。
- 不依赖单次固定文案；可对同一产品、同一店铺多次生成（如不同话术类型、不同语言），结果一致可预期、可对比优化。

---

## 六、接入 LLM 时的使用方式

1. **市调**：保持不变，`runScriptResearch(userInput, storeContext)` 产出 `ScriptResearchContext`。
2. **构建 LLM 输入**：  
   - 系统提示：`buildLLMSystemPrompt(research)`  
   - 用户消息：`buildLLMUserMessage(research)`（即市调摘要 + 「请根据以上市调摘要生成符合要求的话术」）。
3. **调用 LLM**：将系统提示 + 用户消息发给 DeepSeek/OpenAI 等，得到话术初稿。
4. **综合产出**：将 LLM 初稿传入 `synthesizeScript` 的「初稿」步骤（或新增 `synthesizeScriptFromLLM(research, llmContent)`），后续仍做平台合规检查、数据依据拼接、合规提示，逻辑不变。

这样即可实现「先市调再交互 LLM，最后综合产生话术」，且保持平台合规与可复用话术逻辑。

## 七、文件与接口一览

| 文件 | 职责 |
|------|------|
| `backend/src/rules/platformCompliance.ts` | 各平台违规点与合规规则；合规检查 |
| `backend/src/rules/scriptResearch.ts` | 市调：整合用户输入 + 店铺数据 + 平台合规 + 品类实践；LLM 提示构建 |
| `backend/src/rules/scriptSynthesis.ts` | 综合产出：初稿 + 合规检查 + 数据依据拼接 + 合规提示 |
| `backend/src/routes/ai.ts` POST `/script` | 拉取店铺数据 → 市调 → 综合产出 → 返回话术 |
