# 各模块步骤总结

## 📋 总体原则

1. **一个模块一个模块来**：只做当前模块的调试与测试，不做其他模块
2. **先调试后测试**：先把该模块功能调通（修 bug、确认接口与数据），再跑该模块的测试循环
3. **通过再下一模块**：当前模块测试循环全部通过后，再进入下一个模块

## 🔄 标准流程（必须按此闭环执行）

**流程图已固化为 Cursor 规则**（`.cursor/rules/module-test-workflow.mdc`），做模块测试时 Agent 将按此顺序执行，无需重复说明。

```
1. 按模块实现/确认后端功能代码
        ↓
2. 调用 MCP 测试前端，需对标当前模块的市场成品方案
        ↓
3. 判断是否存在问题
   是 → 回到 1（修后端/相关代码后重新 MCP 测）
   否 ↓
4. 输出模块报告  →  5. 人工终测
```

- **详细说明与检查清单**：见 **《模块测试流程-前后端闭环》**（`docs/模块测试流程-前后端闭环.md`）。
- **多轮循环**：若某轮有失败，则「调试 → 自验 → 再测」重复直到通过。详见 **《测试调试多轮循环优化》**（`docs/测试调试多轮循环优化.md`），每轮可用 **《每轮检查清单-模板》**（`docs/每轮检查清单-模板.md`）。
- **前后端测试与评估对标**：见 **《前后端测试与评估对标报告》**（`docs/归档/历史测试报告/前后端测试与评估对标报告.md`），含后端 API 自动化测试结果及模块 1～3 对标评估。
- **抖店直播数据指标对标**：见 **《抖店直播数据项目对标》**（`docs/抖店直播数据项目对标.md`），含核心 6 项、人均观看时长/GPM 等派生指标及**时效（每小时成交额）**说明。

### 不同商店间信息隔离（已实现）

## 📝 模块 3：数据筛选（Dashboard 看板） ✅ 人工终测合格

### 对标样本
- **淘宝/京东商家后台数据看板**：指标勾选、时间范围、实时生效
- **行业共性**：筛选不刷新整页、选项与展示一致、可收起/展开

### 已修复（逻辑与无效按钮）
- **时间周期生效**：前端将 `timePeriod` 传入 `useLiveStats`，请求 `/stats/live?timeRange=week` 等；后端按**自然周期**计算日期范围：**本周**=自然周（周一～周日）、**本月**=自然月、**本季度**=自然季度、**本年**=自然年（当年 1 月 1 日～12 月 31 日），符合用户习惯（非「过去 7 天/过去 30 天」）。日期输出使用**本地日期**（YYYY-MM-DD），避免 UTC 导致「本月」等范围错误。
- **数据周期展示**：统计卡片区域已显示「数据周期：本周」等当前选中周期标签。
- **至少保留一项指标**：取消勾选到最后一项时弹出 Toast「请至少保留一项数据指标」，并保持该项选中（Dashboard 已接入 `useToast`）。
- **空状态提示**：未勾选任何数据项目时展示「请至少选择一项数据指标（在筛选面板中勾选）」。
- **自定义周期**：支持「自定义」可选日期范围（从/至），周期标签显示日期区间；选择月份/选择年份均已支持。

### 自定义日期与数据展示

- **表格日期 → 数据库**：导入时按 **Excel 行中的日期列** 分组，**按日写入 stats**（每出现一个日期就写一条 `storeId + date` 记录）。日期支持 `YYYY-MM-DD`、`YYYY/MM/DD`、`YYYY年M月D日` 等格式；无有效日期时按「当天」一条写入。
- **自定义范围逻辑**：选择「自定义」起止日期（如 2025-12-15～2025-12-21）时，后端按 **date ≥ dateFrom 且 date ≤ dateTo** 汇总，即**所选日期区间内的精确合计**（仅包含区间内各日的 stats，与表格中的日期一一对应）。
- **前端说明**：自定义周期下仪表盘会显示「按所选日期区间精确汇总」，表示数据与所选日期匹配。
- **若自定义仍显示「该时间段暂无数据」**：请确认（1）表格中有日期列且格式可解析；（2）重新导入后该店铺在 `db:inspect` 中有对应日期的 stats；（3）所选区间与导入数据的日期有交集。导入成功后接口缓存已清除，仪表盘会刷新。
- **选择月份 / 选择年份**：与本周、本年、自定义一致，均按 **date ≥ dateFrom 且 date ≤ dateTo** 汇总该区间内所有按日写入的 stats。若「选择月份」显示异常（如时长明显偏大），请先**重新构建后端**（`cd backend && npm run build`）并**重新上传 Excel**，确保导入时「按日期范围整体删除旧 stats 再按日写入」的逻辑生效，避免旧整月数据与新按日数据并存导致重复汇总。
- **上传时间范围**：用户上传的直播明细可以是**自定义几天、一周、整月、季度或自定义区间**；导入只删除**本批表格覆盖的日期区间**及**涉及月份的月初日**（如 2025-12-01），再按日写入，不删未覆盖的日期，避免按周/按几天上传时误删其他天。
- **去重机制（检测到多组重复时仅保留一组）**：
  1. **导入时**：删除本批日期范围及涉及月份的月初日，再按日写入；同日期内按**直播开始时间**（开播时间）唯一去重（有则用该列，无则用直播ID），同一场次只保留一条；写入后对该店铺该范围做同店同日去重（只保留最新一条）。不同店铺（storeId）互不影响。
  2. **同店同日多条**：`npm run db:clean-stats` 清洗「同店同日」重复，**只保留最新一条**（按 createdAt 降序），其余删除。
  3. **月初行疑似整月汇总**：`npm run db:clean-monthly-orphan` 针对**任意月份**（12 月仅为举例）：仅当某月 1 日的 GMV/时长与该月**其余日合计**的比值在 [0.5, 1.5] 时，判定为旧「整月一行」与按日数据并存，**只删除该月初行**；合法单日 1 号数据（比值不在该区间）**不会删除**。stats 表为按日汇总、无开播时间字段，故此处用数值比值判定。
- **成交金额/时长显示两倍**：多为库内残留「整月一行」与按日写入并存。（1）先执行 **`npm run db:clean-monthly-orphan`**（仅删除疑似整月汇总的月初行）；（2）或 **`npm run db:clean-stats`** 清洗同店同日重复；（3）重新上传：导入会删除本批日期范围及涉及月份的月初日，再按日写入。
- **数据仍不对时**：（1）**总互动数恒为 0**：确认 Excel 有「总互动/互动数」列，或「点赞」「评论」「分享」列（系统会汇总为总互动）；（2）**直播时长异常**：确认表格有「直播时长/时长(秒)/时长(分钟)/观看时长」等列且单位正确（秒会按 60 换算为分钟再转小时）；（3）用 `npm run db:inspect` 查看该店铺 stats 条数、日期分布及最近几条示例，与所选周期对照。
- **扩展指标（点赞/评论/分享等）导入后仍为 0**：解析器已扩展列名映射并支持表头首尾空格；支持的列名与排查步骤（含 `DEBUG_EXCEL_HEADERS`）见 **《数据维度扩展-重新导入指南》**；重新导入后验证步骤见该文档「继续验证」小节。

  1. **后端** `POST /api/ai/script`：接受 `storeId`，拉取店铺信息（名称、平台、区域、目标人群、价格带）与品类、近期统计（若有）；生成内容包含【基于店铺】店铺名（平台）、品类、价格带、目标人群，若有转化率则加【数据参考】建议；开场白/主要内容/结束语中嵌入店铺名、人群、品类要点。
  2. **前端**：话术生成/创建脚本均传 `selectedStore?.id`；未选店铺时 Toast「请先选择店铺，以便基于店铺数据生成话术」并中止请求；结果区标题展示「生成的脚本 · 基于店铺「XXX」」、说明改为「以上内容基于所选店铺的品类、人群与近期数据生成…」、增加「基于店铺：XXX」标签。
- **验证**：选店铺后点击「话术生成」，结果应含店铺名、品类、目标人群、数据参考（有统计时），且界面标明基于店铺与数据来源。

### 模块 7 话术生成规则优化（基于市场调研）

- **调研依据**：`docs/直播带货主播话术市场调研.md`（含行业规范、学术研究、转化机制、心理学应用、时长语速、多语言等）
- **规则位置**：`backend/src/rules/scriptGeneration.ts`（话术类型元数据、合规配置、心理学策略库）、`backend/src/rules/scriptPrompts.ts`（话术内容生成函数）
- **详细说明**：`docs/话术生成规则说明.md`

**核心改进**：

1. **话术类型体系**（四类）：人群互动（interaction）、场景化塑品（scenario）、促销活动（promotion）、逼单技巧（closing），对标行业「认识·认同·认购」三分法及实战五分法，每类含核心要素（如场景化的「场景代入→痛点→解决方案→效果→CTA」）。
2. **时长与语速规范**：20～60 秒、180～220 字/分、前 30 秒黄金开场、前 3 分钟留人期、单次连续表达不超过 15 秒、每 3 句话穿插 1 次互动，符合行业最佳实践。
3. **心理学策略**：锚定、从众、损失厌恶、互惠、权威、稀缺、FOMO 等已嵌入促销与逼单类话术模板（如「原价 599，直播间 199」「已有 127 位家人下单」「明天恢复原价」等）。
4. **合规规则**：禁用词分类（医疗、绝对化、虚假、夸大），高风险类目（保健食品、医疗器械、酒类等），生成后自动调用 `checkCompliance`，有警告则追加「⚠️ 合规提示」。
5. **多语言与本地化**：中文（含「家人们」「小黄车」等）、英文（国际化表达）、泰语（基础版，含问候与本地表达），每语种独立设计系统提示词与话术结构，不仅是翻译。
6. **入参扩展**：新增 `productName`（必填）、`price`、`features`、`targetAudience`、`scriptType`（默认 interaction）、`language`（默认 zh-CN），兼容旧版 `topic`、`duration`、`style`、`storeId`。
7. **生成逻辑**：有 `productName` → 调用 `buildScriptContent(scriptType, language, params)` 生成专业话术；无 `productName` → 使用店铺 + topic 生成通用大纲（兼容旧版）。

**验证**：选店铺后，在话术生成/创建脚本页填写产品名称、选择话术类型（如「场景化塑品话术」）与语言（如「中文」），点击「生成」，应返回符合该类型结构（场景代入、痛点、解决方案、效果、CTA）的话术，且包含店铺名、品类、转化率等数据参考（如有），不再是通用「开场白+主要内容+结束语」模板。

**v2.0 质量升级**（2025-02-02）：
- **问题**：生成的话术内容"太低质"，还是模板化示例（如"早起不想化妆？久坐背痛？"），未根据具体产品（如"猫宠-不锈钢猫碗"）生成针对性话术。
- **优化**：重写 `scriptPrompts.ts`，实现**智能场景识别与话术定制**：
  1. 根据产品名称关键词（宠物、美妆、食品、家居等）智能识别产品类型，生成对应场景的具体话术（如宠物产品 → "家里主子的碗总是不干净，塑料的容易滋生细菌"）。
  2. 自动计算价格锚定（原价 = 输入价格 × 2.5）、优惠金额、折扣比例，融入话术。
  3. 嵌入真实数据营造从众效应（"已有137位家人下单""5000+好评"）。
  4. 口语化表达，使用"家人们""扣1""手慢无""小黄车"等直播术语。
  5. 融入心理学策略（锚定、从众、FOMO、损失厌恶等），话术结构更具转化力。
- **验证**：用相同输入（产品：猫宠，价格：299，特点：不锈钢，类型：场景化塑品）再次生成，应返回针对宠物碗场景的具体话术（如"塑料碗发黄发臭，陶瓷碗易碎"等真实痛点），而非通用化妆品示例，且话术可直接在直播间使用。

**完整销售流程话术（2025-02-02新增）**：
- **类型标识**：`full-sales`，在话术类型下拉框中显示为"完整销售流程话术（推荐）"
- **设计理念**：基于用户提供的完整转化漏斗逻辑（圈人群→塑品→打消顾虑→利益点→售后→逼单），这是一套经过实战验证的单品推销流程。
- **六步结构**：
  1. **圈人群+塑品**（20-30秒）：圈定目标人群，激发美好生活需求或现实不满需求
  2. **卖点提炼**（25-35秒）：核心卖点（高意向用户，建议做成手卡/KT板）+ 次要卖点1、2、3（中低意向用户）
  3. **打消顾虑**（20-30秒）：互动环节，引导出客户下单前顾虑（SKU选择、质量安全等）并点对点解决
  4. **利益点**（20-30秒）：赠品组套（买一送三）、折扣力度（自动计算原价、优惠金额、折扣比例）、大促活动（今晚12点结束）
  5. **售后保障**（10-15秒）：7天无理由退换、质量问题包换、假一赔十、48小时发货、专属客服（售后小卡片）
  6. **逼单**（15-20秒）：重申核心卖点+营销利益点+售后，强调库存稀缺性（剩XX件、已卖XX件、现在XX人在看），倒计时话术（3-2-1下单）
- **智能生成**：根据产品类型（宠物、美妆、食品、服装、家居、数码等）自动识别并生成对应场景的话术，价格自动计算锚定价（× 2.8）、优惠金额、折扣比例。
- **可循环**：完整版 90-180 秒，简化版循环 30-40 秒（可循环播放2-3次），也可拆分使用各环节。
- **适用场景**：高价值商品（客单价 > 100 元）、需要深度讲解的产品、精细化运营、直播间人气稳定。
- **使用建议**：核心卖点视觉化（手卡/KT板）、结合商品评论分析（提示词：用客户容易理解的语言帮我提炼XX的核心卖点）、实时回复公屏、调整营销活动。
- **详细说明**：见 `docs/完整销售流程话术说明.md`

**话术流程升级（先市调再综合产出，避免一次性话术）**：
- **流程**：用户输入 + 店铺数据 → **市调**（整合平台合规 + 品类最佳实践 + 数据依据）→ **综合产出**（模板/LLM 初稿 + 平台合规检查 + 数据依据拼接）→ 最终话术。
- **平台违规点**：各平台（抖音、淘宝、快手、TikTok、微信、通用）独立配置禁用词、慎用词、违规类型与话术风格建议；品类可叠加额外约束（`backend/src/rules/platformCompliance.ts`）。
- **从用户角度**：需求激发（美好生活/现实不满）、卖点分层（高意向/中低意向）、打消顾虑、利益点、售后、逼单，形成可复用、实测有效的话术逻辑。
- **LLM 接入**：市调产出「市调摘要」与「系统提示/用户消息」接口（`scriptResearch.ts`），接入 DeepSeek 等后可先市调再交互 LLM，最后在综合产出层做合规与拼接。
- **目标人群解析**：用户输入的「目标人群」不再直接进话术。先做解析（`scriptAudienceAnalysis.ts`），产出「称呼」与「痛点/需求」再用于生成。例如「资深中产、多猫家庭」→ 称呼「家里好几只主子的铲屎官们」、痛点「多只猫的卫生和空间、笼子要耐用又好清洁」，话术中用「家里好几只主子的铲屎官们，有没有正在为……发愁的？」而非「有多少资深中产、多猫家庭遇到过这个问题？」。
- **规则编码检查**：`docs/话术规则编码检查清单.md` 逐项对照「避免一次性话术」等要求，核对市调、综合产出、人群解析、平台合规、话术结构是否均已系统编码。
- **功能代码与配置**：话术规则已写入功能代码（`backend/src/rules/*.ts`），可配置项从 `backend/config/script-rules/*.json` 加载（缺省用代码内默认值）。配置工具：`GET /api/config/script-rules` 读取当前生效配置，`PUT /api/config/script-rules` 写回 JSON（需认证）。
- **详细说明**：见 `docs/话术生成流程-市调与综合产出.md`
