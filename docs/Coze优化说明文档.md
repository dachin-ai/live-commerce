## 系统概述

本系统是基于 LangChain Agent 框架开发的 TikTok 直播运营智能助手，专注于生成高质量、高转化率的直播带货话术，并提供数据洞察、任务管理、节日营销等全链路运营支持。

### 技术栈
- **AI 模型**：豆包大模型 (doubao-seed-2-0-pro-260215)
- **框架**：LangChain Agent
- **编程语言**：Python 3.10+
- **推理能力**：High（深度思考模式）

### 系统特点
✅ 支持8个国家本地化运营
✅ 自动识别任务类型，智能路由
✅ 内置质量控制，话术符合7大核心标准
✅ 支持竞品分析和打击策略
✅ 话术效果追踪与优化

## 话术生成功能

> ⚠️ v3.0 新增：前端现已支持传递 `price_level`（价格定位）和 `product_role`（产品角色）参数，用于弱塑品/强塑品话术模式自动路由。以下保留原有全部规范。

### 功能定位
生成高质量、高转化率的直播带货话术，支持8个国家本地化运营。

### 核心标准（7大特征）

#### 1. 场景化设计
- **要求**：使用本地化场景唤醒用户需求
- **示例**：
  - 泰国：曼谷堵车、健身房远、炎热天气
  - 印尼：雅加达炎热、MPASI辅食、家庭聚会
  - 马来西亚：吉隆坡交通、多元文化饮食

#### 2. 痛点挖掘
- **要求**：每个环节都有明确的痛点，层层递进
- **痛点类型**：时间、空间、价格、体验、效果

#### 3. 可视化设计
- **要求**：小白主播提示必须明确，重点展示具体功能
- **动作标注**：**(右手拿起炒锅展示)**、**(右手指向评论区)**
- **粗体标注**：**1-18档调速**、**15档坡度调节**

#### 4. 数据对比
- **要求**：使用具体数字和价格对比，凸显性价比

#### 5. 本地化表达
- **要求**：使用正确货币符号、本地化代词、本地化场景
- **货币符号**：泰国 ฿、印尼 Rp、马来西亚 RM、菲律宾 ₱、新加坡 S$、越南 ₫、中国 ¥、美国 $

#### 6. 紧迫感设计
- **要求**：有库存限制、限时优惠、促单话术

#### 7. 行动指引
- **要求**：明确的操作步骤、促单结尾、行动动词

### 话术类型

| 类型 | 说明 | 时长 | 适用场景 |
|------|------|------|----------|
| **full_process** | 完整流程话术 | 6-8分钟 | 新品推广、专场活动 |
| **interaction** | 互动话术 | 1-2分钟 | 活跃气氛、提升互动 |
| **scenario** | 场景话术 | 2-3分钟 | 场景化推广 |
| **promotion** | 促销话术 | 2-3分钟 | 大促活动、清仓 |
| **closing** | 逼单话术 | 1-2分钟 | 活动尾声、库存告急 |
| **simple_cycle** | 一卖点一逼单话术 | 2分钟/循环 | 引流款、爆单款、低客单价 |

### 价格参数区分功能

#### 弱塑品强营销模式
- **适用条件**：price_level为"低"或"中" + product_role为"引流款"或"爆单款"
- **特点**：塑品30-45秒，快速切入价格，高频互动，2分钟循环

#### 强塑品理性说服模式
- **适用条件**：其他情况（高价、利润款、战略款等）
- **特点**：塑品90-120秒，Before/After对比，算账法，温和理性

### 印尼市场深度优化规则

#### 文化禁忌规则（最高优先级）
- **禁止使用左手操作**：所有展示、递交、指向动作必须且只能使用右手
- **禁止咆哮式逼单**：采用温和、真诚、节奏稳健的沟通方式
- **柔性植入Halal心智**：柔性强调"洁净无污染"标准

#### Bunda话术风格规则
- 使用"Bunda"（妈妈们）、"姐妹们"、"家人们"
- 亲切、专业、温暖的语气

### 输出限制（⚠️ 必须严格遵守）
- **只输出中文**：禁止生成任何非中文内容
- **禁止输出图表**：绝对禁止输出 Mermaid、ECharts 等图表代码
- **禁止输出分析报告**：绝对禁止输出分析报告内容
- **禁止输出表格**：绝对禁止输出 Markdown 表格或 HTML 表格

## 待办事项生成功能

> ⚠️⚠️⚠️ **本节有重大变更，请仔细阅读！**
> 输出格式、字段名、描述结构均已与前端系统严格对齐。

### 功能定位
生成结构化、可执行的运营任务清单，帮助运营团队拆解目标、规划工作。

### 任务类型识别
当用户请求中包含「待办」、「任务」、「todo」、「task」、「智能生成」、「生成待办」等关键词时，为待办生成请求。

### 生成策略
- **优先直接生成**：直接流式生成纯JSON，不调用工具（更快、统一输出格式）

---

### ⚠️ 输出格式要求（v3.0 重大变更）

#### 必须严格遵守的 JSON 格式

```json
{
  "tasks": [
    {
      "title": "简洁的任务标题（不超过20个字）",
      "description": "【目标】...\n【数据来源】...\n【执行步骤】...\n【参数配置】...\n【验证方案】...",
      "priority": "urgent",
      "estimated_days": "3天",
      "category": "core",
      "responsible": "运营"
    }
  ]
}
```

#### ⚠️ 字段名强制规范（禁止使用其他字段名）

| 字段 | 类型 | 必须 | 说明 | ⛔ 禁止使用 |
|------|------|------|------|------------|
| **title** | string | ✅ | 任务标题 | ⛔ 禁止用 task_name、name、task |
| **description** | string | ✅ | 任务描述（按5段结构） | - |
| **priority** | string | ✅ | 只能是 `"urgent"` 或 `"normal"` | ⛔ 禁止用 高/中/低、high/low |
| **estimated_days** | string | ❌ | 预计天数，如"3天"、"7天" | ⛔ 禁止用 estimated_time、deadline |
| **category** | string | ❌ | 分类：`analysis`、`core`、`strategy` | ⛔ 禁止用 tags |
| **responsible** | string | ❌ | 责任人，如"运营"、"主播+运营" | - |

#### ⚠️ 包裹格式强制规范

- ✅ 正确：`{"tasks": [...]}`（必须包裹在 tasks 对象中）
- ⛔ 错误：直接返回数组 `[{...}, {...}]`
- ⛔ 错误：添加 markdown 代码围栏 ` ```json ... ``` `
- ⛔ 错误：在 JSON 前后添加任何文字说明

#### ⚠️ priority 值强制规范

- ✅ 正确：`"priority": "urgent"` 或 `"priority": "normal"`
- ⛔ 错误：`"priority": "高"` / `"priority": "中"` / `"priority": "低"`
- ⛔ 错误：`"priority": "high"` / `"priority": "medium"` / `"priority": "low"`

### 任务描述结构（5段式，必须严格遵守）

每个任务的 `description` 字段必须严格包含以下 5 个段落，标题使用中文全角书名号括号形式 `【xxx】`：

#### 1)【目标】— SMART目标，含时间边界
```
【目标】7天内将直播间点击率从3%提升至5%，覆盖全部直播时段
```

#### 2)【数据来源】— 当前值 / 目标值 / 依据说明
```
【数据来源】当前点击率3.2%（近30天均值），平台同品类基准5.1%，差距1.9个百分点
```

#### 3)【执行步骤】— 每步含动作 + 截止时间 + 责任人
```
【执行步骤】
第一步：分析近30天封面数据，筛选点击率TOP3模板（4月21日前/运营）
第二步：基于TOP模板设计3组新封面（4月22日前/设计）
第三步：A/B测试新封面，每组运行2天（4月24日前/运营）
第四步：确定最优封面并全量应用（4月25日前/运营）
```

#### 4)【参数配置】— 关键参数 + 来源 + 调整方案
```
【参数配置】封面尺寸1080×1920px；A/B测试每组最低曝光500次；点击率阈值5%
```

#### 5)【验证方案】— 指标 / 方法 / 周期 / 成功标准 / 失败处理
```
【验证方案】7天后对比A/B测试数据，成功标准：点击率≥5%；若未达标则更换配色方案重新测试
```

#### 可选追加：6)【资源需求】
```
【资源需求】设计师1人/天，测试预算200元流量券
```

- 如果 `existing_tasks` 中已存在某任务标题，禁止生成标题相同的任务
- 相似任务合并为更高级别的任务
- 每个任务聚焦一个核心问题

### JSON 输出质量要求
- **禁止**在字符串值内使用未转义的英文双引号，价格举例用「」包裹

- **禁止**使用单引号替代双引号
- **必须**每条任务严格用英文双引号与冒号
- **必须**直接流式输出 JSON，不添加任何 markdown 围栏或前后文字

## 数据分析功能

### 功能定位
提供电商运营数据分析和优化建议，帮助运营团队发现问题、制定策略。

### 分析类型
1. **直播数据分析**：观看人数、互动率、转化率、GMV → 数据洞察 + 优化建议
2. **竞品数据分析**：竞品价格、卖点、销量、评价 → 对比分析 + 打击策略
3. **销售数据分析**：销售额、客单价、复购率 → 趋势分析 + 增长建议

### 识别规则
- 包含「分析」、「对比」、「复盘」等关键词
- 且未使用【product_name】等标签（否则优先匹配话术生成）

### 输出格式
- 文本格式，包含数据洞察和优化建议
- 不输出图表或表格

提供节日营销策略和活动规划，帮助运营团队抓住节日流量红利。

- **印尼**：斋月、开斋节、独立日
- **泰国**：宋干节、水灯节
- **马来西亚**：开斋节、农历新年

- **越南**：春节、中秋节
- **新加坡**：春节、开斋节、圣诞节

### 节日营销策略要素
1. **节日氛围营造**：视觉元素、关键词、消费场景
2. **节日专属优惠**：限时优惠、节日套餐、定制赠品
3. **节日话术优化**：融入文化、祝福语、节日场景
4. **节日投放优化**：提前预热(7-14天)、节中加码、节后返场

## 常见问题

### Q1: 待办生成的 JSON 格式是什么？
**A**: 必须输出 `{"tasks":[...]}` 格式，每个任务包含 `title`、`description`、`priority` 三个必需字段和 `estimated_days`、`category`、`responsible` 三个可选字段。`priority` 只能是 `"urgent"` 或 `"normal"`。

### Q2: 待办的 description 必须怎么写？
**A**: 必须按5段结构：【目标】→【数据来源】→【执行步骤】→【参数配置】→【验证方案】，可选追加【资源需求】。

### Q3: 如何区分话术生成和数据分析？
**A**: 优先级规则：话术生成 > 待办生成 > 数据分析。如果包含【product_name】，优先生成话术。

### Q4: 印尼市场话术有哪些特殊要求？
**A**: 右手操作、温和逼单、Bunda称呼、本地化关键词（Ready Stock、Anti Lengket、MPASI等）、话术降维。

### Q5: store_data 的格式是什么？
**A**: 必须是完整的 JSON 字符串，包含 conversion_rate、daily_views 等业务数据字段。详见「store_data 字段说明」章节。

### C. 点击率定义说明（当前系统可用指标）

系统目前主要基于 TikTok/抖音后台导出的 Excel 数据，可计算的点击率类型如下：

| 点击率类型 | 计算公式 | 数据来源 | 系统是否可用 | 说明 |
|-----------|---------|---------|:---:|------|
| **直播商品点击率** | 商品点击数 ÷ 商品曝光数 × 100% | Excel 导入 `productClicks` / `productViews` | ✅ 可用 | 直播间内商品卡的点击转化效率 |
| **看播→点击转化率** | Excel 原始字段 `clickThroughRate` | Excel 导入 | ✅ 可用 | TikTok 后台定义的观看→点击商品转化率 |
| **视频封面点击率** | 封面点击 ÷ feed 曝光 × 100% | 需 TikTok API 或独立报表 | ❌ 暂不可用 | 短视频/直播封面在信息流中的点击吸引力 |
| **店铺商品点击率** | 店铺商品详情页点击 ÷ 店铺商品曝光 × 100% | 需电商后台独立报表 | ❌ 暂不可用 | 店铺场景下的商品页面点击效率 |
| **直播间封面点击率** | 直播间入场 ÷ 直播封面曝光 × 100% | 需 TikTok 流量分析报表 | ❌ 暂不可用 | 直播间在推荐/关注页的进入率 |

> **注意**：标记为「暂不可用」的指标需要从 TikTok API 或额外的 Excel 报表类型中获取。当前系统 Excel 导入仅覆盖「直播数据明细」一种报表，不包含这些维度的独立指标。后续如接入 TikTok Open API 或新增数据报表类型，可扩展支持。

#### 待办生成中传递给 LLM 的点击率数据

store_data 中包含以下字段，LLM 可据此生成点击率优化相关待办：
- `total_product_views` — 商品曝光数
- `total_product_clicks` — 商品点击数
- LLM 可自行计算：商品点击率 = `total_product_clicks / total_product_views × 100%`
