# 待办多语言与内置翻译说明

## 设计原则

> **生成时固定单语（zh-CN）+ 内置翻译按需提供多语言**  
> 避免为多语言重复调用 LLM、浪费 API、拉长响应；功能好用比操作繁琐更重要。

---

## 1. 生成时只写一种语言

### 后端行为

- **POST /api/ai/generate-tasks**：无论前端传入什么 `locale`，后端在调用 `generateIntelligentTodosWithLLM` 时**强制 `locale: 'zh-CN'`、`countryCode: 'CN'`**。
- LLM 始终用中文输出任务的 `title` 和 `description`。
- 写入数据库时：`title`、`description` 为中文，`title_i18n`、`description_i18n` 初始为空或 NULL。

### 优势

- **单次生成、响应更快**：不需要在 LLM prompt 中要求多语言，避免输出冗长和额外 token 消耗。
- **减少 API 浪费**：不会为同一批待办多次调用 LLM 只为换语言。
- **少点一次生成**：用户切换语言时不用再点「智能生成」。

---

## 2. 内置翻译（不依赖 LLM）

### 实现

- **翻译引擎**：`google-translate-api-x`（Google Translate 非官方接口，免费、无需 API key）。
- **后端工具**：`backend/src/utils/translate.ts` → `translateText(text, targetLang, sourceLang)`。
- **内存缓存**：同一段文本 + 语言对会缓存在 Map（最多 500 条），重复翻译不会再请求外部 API。

### API 端点

#### POST /api/translate

- **用途**：单个文本翻译（通用工具）。
- **Body**：`{ text: string, targetLang: string, sourceLang?: string }`  
  - `targetLang` / `sourceLang`：支持 `zh-CN`、`en-US`、`th-TH`  
  - 内部会映射为 `zh-CN` / `en` / `th` 供 Google Translate 使用
- **返回**：`{ translatedText: string }`

#### POST /api/tasks/translate-for-locale

- **用途**：批量翻译当前用户（可选店铺）下所有待办任务，并写入 `title_i18n` / `description_i18n`。
- **Body**：`{ storeId?: string, locale: string }`
- **行为**：
  1. 查询当前用户的待办任务（`status = 'pending'`）。
  2. 对每条任务，检查 `title_i18n[locale]` 是否已存在：
     - **已有**：跳过（不重复翻译，避免浪费）。
     - **没有**：调用 `translateText` 翻译 `title` 和 `description`，将结果写入 `title_i18n[locale]` / `description_i18n[locale]`。
  3. 若翻译过程中触发限额或网络错误，会**立即停止**并返回已成功翻译的数量 + `error: 'QUOTA_EXCEEDED'`。
- **返回**：`{ translated: number, total: number, error?: 'QUOTA_EXCEEDED' }`

## 3. 前端行为

### 自动翻译

- **触发条件**：当界面语言不是 `zh-CN`（如 `en-US` 或 `th-TH`）且存在**尚未翻译到当前语言**的待办时。
- **执行时机**：`TaskList` 组件用 `useEffect` 监听 `currentLocale`、`needsTranslate`、`selectedStore?.id` 变化，满足条件时**自动调用一次** `handleTranslateToCurrentLanguage()`。
- **防重复**：用 `lastAutoTranslateLocaleRef` 记录已自动翻译的语言，避免同一语言下重复触发。
- **结果**：翻译成功后刷新任务列表，已翻译的待办立即显示为目标语言；若触发限额则给出提示，部分翻译成功的仍会显示。

### 显示逻辑

```ts
// 优先用缓存，否则原文
getTaskDisplayTitle(task, currentLocale) 
  = task.title_i18n?.[currentLocale] || task.title

getTaskDisplayDescription(task, currentLocale) 
  = task.description_i18n?.[currentLocale] || task.description
```

- 当 `title_i18n` 为数据库 TEXT 字符串（如 `'{"en-US":"..."}'`）时，`parseTaskI18n` 会自动解析为对象。
- 切换语言时，组件会取对应 locale 的缓存；没有则显示原文（中文）。

### 手动翻译按钮

- **按钮文案**：`t('tasks.translateToCurrent')`（zh: "显示为当前语言", en: "Show in current language", th: "แสดงเป็นภาษาปัจจุบัน"）。
- **显示条件**：`needsTranslate` 为 true（即有未翻译的待办 + 当前语言非中文）。
- **行为**：点击后手动调用 `translateTasksForLocale` 并刷新；若自动翻译已触发但用户想重试，可点此按钮。

## 4. 每日限额问题

### Google Translate 非官方接口

- **优势**：无需 API key、免费、相比 MyMemory 限额更宽松。
- **限制**：仍有频率限制（具体额度取决于 IP 与请求频次），但通常能满足中小型应用日常使用。
- **风险**：非官方接口可能随时变更或失效（概率较低，`google-translate-api-x` 是活跃维护的项目）。

### 限额触发时的行为

1. **后端**：捕获错误 → 标记 `error: 'QUOTA_EXCEEDED'` → 返回已翻译数量。
2. **前端**：显示 Toast 提示：
   - `今日免费翻译额度已用完，待办将显示原文。请明日再试或切换为中文界面。`（或对应语言）
3. **降级**：未翻译的待办继续显示中文，不影响功能使用；已翻译的正常显示目标语言。

### 减少额度消耗的措施

1. **缓存**：同一段文本 + 语言对缓存在内存（最多 500 条），重复翻译不会再请求外部。
2. **按需翻译**：仅当用户切换到非中文语言时才触发翻译，不会在生成时批量写多语言。
3. **增量翻译**：已有 `title_i18n[locale]` 的任务跳过，只翻译缺失的部分。

### 升级方案（可选）

若免费额度仍不够，可考虑：

- **Google Cloud Translation API**（官方，需 API key，按量付费，稳定）：在 `.env` 中配置 `GOOGLE_TRANSLATE_API_KEY`，并修改 `backend/src/utils/translate.ts` 切换为官方 SDK。
- **自建翻译服务**：部署 LibreTranslate（开源、自托管、无限额）。
- **缓存到数据库**：将翻译结果存入 PostgreSQL 持久化缓存表（跨重启保留），进一步减少外部请求。

## 5. 数据库结构

### tasks 表新增字段

| 字段            | 类型 | 说明                                         |
|-----------------|------|----------------------------------------------|
| `title_i18n`    | TEXT | JSON 字符串，如 `{"en-US":"...","th-TH":"..."}` |
| `description_i18n` | TEXT | 同上，存储描述的多语言缓存                   |

- **迁移**：在 `backend/src/db.ts` 的 `initDatabase()` 中通过 `ALTER TABLE tasks ADD COLUMN title_i18n TEXT` 实现，兼容已有数据库（自动检测重复列）。
- **初始值**：新生成的待办这两个字段为 NULL 或空 JSON `{}`；翻译后写入对应 locale 的翻译。

## 6. 用户体验流程

1. **首次使用（中文）**  
   → 点击「智能生成」 → 后端返回中文待办 → 显示中文。

2. **切换为英文**  
   → 界面语言切为 EN → TaskList 检测到 `needsTranslate = true` → **自动**调用 `POST /api/tasks/translate-for-locale` 将所有待办翻译为 `en-US` 并缓存 → 刷新列表 → 显示英文标题与描述。

3. **再次切换为泰文**  
   → 检测到待办未有 `th-TH` 缓存 → **自动翻译** → 显示泰文。

4. **切回中文**  
   → 显示原文（无需翻译），且已有的 `en-US`、`th-TH` 缓存保留；下次切换时直接用缓存，不再请求翻译 API。

5. **限额用完时**  
   → Toast 提示「今日免费翻译额度已用完…请明日再试或切换为中文」 → 已翻译的正常显示，未翻译的显示中文 → 用户可切回中文继续操作，或明日再切换到其他语言。

## 7. 对比「旧方案」与「新方案」

| 对比项             | 旧方案（依赖 LLM 多语言输出）      | 新方案（内置翻译）                   |
|--------------------|------------------------------------|------------------------------------|
| **生成时间**       | LLM 需输出多语言，响应慢            | 固定 zh-CN，响应快                  |
| **Token 消耗**     | 每次生成都可能要求多语言，消耗多    | 仅用中文生成，省 token              |
| **切换语言**       | 可能需再点「生成」获取不同语言版本  | **自动翻译 + 缓存**，无需重新生成   |
| **每日限额**       | 无限额（但消耗更多 LLM 配额）      | Google Translate 非官方免费接口，通常够用，触发限额时有明确提示与降级 |
| **多次点击生成**   | 易重复调 LLM                        | 只生成一次 zh-CN，其他语言用翻译    |

## 9. 常见问题

### Q1: 为什么待办仍显示中文？

**A1**：可能原因与解决方案：
1. **首次切换语言时还未翻译**  
   → 自动翻译会在组件加载后自动触发（约 1-3 秒），请稍等或刷新页面；也可点击「显示为当前语言」按钮手动触发。
2. **网络问题或翻译接口失败**  
   → 查看浏览器控制台或后端日志，检查 `/api/tasks/translate-for-locale` 是否返回错误。
3. **触发每日限额**  
   → 会有 Toast 提示；此时可切换为中文，或明日再试。

### Q2: Google Translate 非官方接口会失效吗？

**A2**：`google-translate-api-x` 是社区维护的项目，理论上可能随 Google 调整而失效，但：
- 该包较活跃，通常能及时适配变更。
- 若担心稳定性，可切换为 **Google Cloud Translation API**（官方，需 API key，按量付费）或 **LibreTranslate**（开源自托管，无限额）。
- 切换方式：修改 `backend/src/utils/translate.ts` 中的实现即可，接口层（`POST /api/translate`、`translate-for-locale`）无需改动。

### Q3: 如何让待办在生成时就有多语言？

**A3**：如果确实需要在生成时就写入多语言（不推荐，浪费 API 且响应慢），可以：
1. 在 `backend/src/routes/ai-refactored.ts` 的 `POST /generate-tasks` 中，恢复传入 `bodyLocale` / `bodyCountryCode` 给 `generateSuggestedTodosForStore`。
2. 生成后立即调用 `translateTasksForLocale` 将生成的待办翻译为所有目标语言（zh-CN、en-US、th-TH）并写入 `title_i18n`。
3. 但这样会：①增加生成耗时（LLM + 多次翻译）、②多耗 API token（如果用 LLM）或翻译请求、③违背「功能好用比操作繁琐更重要」的原则。

### Q4: 我想切换为付费的 Google Cloud Translation API，怎么改？

**A4**：
1. 安装官方 SDK：`npm install @google-cloud/translate`。
2. 在 `.env` 中添加 `GOOGLE_TRANSLATE_API_KEY=your_api_key`（或用服务账号 JSON）。
3. 修改 `backend/src/utils/translate.ts`：
   ```ts
   import { Translate } from '@google-cloud/translate/v2'
   const translateClient = new Translate({ key: process.env.GOOGLE_TRANSLATE_API_KEY })
   
   export async function translateText(text, targetLang, sourceLang) {
     // ... cache check ...
     const [translation] = await translateClient.translate(text, {
       from: toGoogleLang(sourceLang),
       to: toGoogleLang(targetLang),
     })
     // ... cache save ...
     return translation
   }
   ```
4. 官方 API 稳定、无非官方失效风险，但需付费（每月有一定免费额度，超出后按字符计费）。
