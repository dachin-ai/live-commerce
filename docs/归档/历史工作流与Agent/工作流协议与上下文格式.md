# 工作流协议与上下文格式

## 一、工作流引擎位置

- 引擎代码：`backend/src/workflow_engine/`
- 调度器：`scheduler.ts`（基于时间/触发条件分派迭代）
- 角色分发器：`role_dispatcher.ts`（按序调用五角色）
- 上下文管道：`context_pipeline.ts`（跨角色上下文与 state 读写）
- 输出收集器：`output_collector.ts`（汇总产出至 `outputs/第N轮迭代/`）
- 状态管理器：`state_manager.ts`（`data/shared_state/state.db` 去重、游标、检查点）

## 二、五角色执行顺序与产出

| 角色 ID | 角色名 | 产出物 |
|--------|--------|--------|
| planner | 规划师 | 《季度优化主题》+《路线图》（主题.md、路线图.md） |
| industry_expert | 行业专家 | 《行业最佳实践方案》结构化 JSON（行业最佳实践.json） |
| senior_user | 资深用户 | 《用户体验测试报告》+《A/B 测试建议》（用户体验测试报告.md、AB测试建议.md） |
| engineer | 程序工程师 | 《技术实施方案》+《部署包》（技术实施方案.md、部署包/） |
| novice | 小白 | 《疑问库》（疑问库.md） |

角色执行器签名为：`(context: WorkflowContext) => Promise<RoleOutput>`，由 role_dispatcher 按序调用，output_collector 统一落盘并更新 manifest。

## 三、WorkflowContext 结构

```ts
interface WorkflowContext {
  roundId: string          // 如 round_1
  roundLabel: string       // 如 第1轮迭代
  currentRoleIndex: number // 当前执行到的角色下标（0~4）
  previousOutputPaths: Record<string, string>  // 上一角色产出路径，供下游角色读取
  storeId: string | null
  taskIds: string[]
  startedAt: string
  updatedAt: string
}
```

- 由 `context_pipeline.loadContext(roundId, roundLabel, options?)` 从 state 加载/构建。
- 角色完成后通过 `context_pipeline.saveContext(ctx, roleId, status, payload)` 持久化检查点与游标。

## 四、RoleOutput 结构

```ts
interface RoleOutput {
  roleId: WorkflowRoleId
  files: { path: string; content: string }[]
  summary?: string
}
```

- `path` 为相对 `outputs/第N轮迭代/` 的路径；`content` 为文件内容，由 output_collector 写入并写入 manifest。

## 五、触发与 API

- **人工触发**：`POST /api/workflow/trigger`，body 可选 `roundId`、`roundLabel`、`resumeFromRoleIndex`。
- **轮次列表**：`GET /api/workflow/rounds`，返回 `{ roundId, roundLabel }[]`。
- **轮次详情**：`GET /api/workflow/rounds/:roundId`，返回检查点与 manifest。
- 产出静态访问：`GET /api/workflow/outputs/:roundLabel/:filePath`（由 express.static 提供）。

## 六、与参考文档对应关系

- 参考中的「工作流引擎（src/workflow_engine/）」对应本仓库 `backend/src/workflow_engine/`（TypeScript 实现）。
- 「五角色定义」与「自动化闭环流程」按上表与 `role_dispatcher` 顺序实现；迭代周期 4–7 天由后续 scheduler 配置（如 cron）控制。
- 「技术底座与资产位置」：设计文档在 `docs/`，交付产物在 `outputs/`，共享状态在 `data/shared_state/state.db`。
