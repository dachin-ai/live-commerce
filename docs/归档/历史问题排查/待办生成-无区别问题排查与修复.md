# 待办事项区域「生成的待办无区别」排查与修复

## 问题

用户反馈：待办事项区域生成的待办**无区别**（内容雷同、或看起来都像同一类任务）。

## 可能原因（排查结论）

| 可能原因 | 结论 | 处理 |
|----------|------|------|
| **1. 交互提示词不合适** | ✅ 是主因 | 原 prompt 为单行密排（`店:xxx\|本周:xxx\|...`），模型难以解析；且未要求「每条任务不同维度」，易产出雷同句。 |
| **2. 前端/数据整合问题** | ⚠ 部分 | 标题被强制截断为 20 字，多条不同建议截断后可能变成同一条；去重仅按完整 title，未做「语义去重」。 |
| **3. 过程耗时过久** | ⚠ 部分 | 一次性调用默认 25s 超时，Coze 若稍慢易超时 → 返回空 → 走规则兜底，规则任务模板固定，多次生成易雷同。 |

## 已做修改

### 1. LLM 提示词（`generateIntelligentTodosWithLLM`）

- **系统提示**
  - 明确要求：**每条任务必须针对不同维度**（时段/互动/选品/引流/节日/时长/转化等），标题与描述不可重复或雷同。
  - 标题放宽为 ≤30 字、描述 ≤120 字，并禁止泛泛的「提升 X 至 Y%」类模板句。
- **用户提示**
  - 改为**分段结构**：  
    `【店铺】… | 【本周数据】… | 【历史对比】… | 【时间与节日】… | 【阶段与重点】…`
  - 无历史数据时显式写「无历史数据，仅基于本周」，避免模型瞎猜。
  - 结尾强调：**根据以上不同维度生成若干条差异化待办，条数按需**。

### 2. 数据与解析

- **标题/描述长度**
  - 标题由 20 字改为最多 **50 字**，描述由 100 字改为 **200 字**，减少「不同建议被截成同一条」。
- **去重**
  - 在解析 LLM 返回的 `tasks` 时，按**标题前 25 字**做一次去重，避免两条几乎相同的任务都展示。
- **空返回与解析失败**
  - LLM 返回空、未命中 JSON、或 `tasks` 为空/解析异常时，打**明确日志**（`[待办生成] LLM 返回空/未找到合法 JSON/解析失败，将走规则兜底`），便于区分是超时、格式错误还是未配置。

### 3. 超时与前端提示

- **待办生成专用超时**
  - `callLLMOnce` 增加可选参数 `timeoutMs`；待办生成传入 **40 秒**（默认 25 秒仍用于其他场景），降低因超时回退到规则的概率。
- **前端**
  - 智能生成按钮在请求中展示：「生成中…(约 20–40 秒)」，避免用户过早刷新或认为卡死。

## 涉及文件

- `backend/src/routes/ai-refactored.ts`：`generateIntelligentTodosWithLLM` 的 prompt、解析、去重与日志。
- `backend/src/services/scriptLLM.ts`：`ScriptLLMOptions.timeoutMs`，Coze 一次性调用使用该超时。
- `frontend/src/components/TaskList.tsx`、`frontend/src/components/AIFeatures.tsx`：等待提示文案。

## 如何验证

1. **看后端日志**
   - 若经常出现 `[待办生成] LLM 返回空（可能超时或未配置）` → 检查 Coze 配置与网络，或适当再提高 `timeoutMs`。
   - 若出现 `未找到合法 JSON` 或 `JSON 解析失败` → 多为模型未按 JSON 输出，可再收紧 system 中的格式说明或换模型。
2. **看返回 metadata**
   - 接口返回 `metadata.llmIntelligentCount > 0` 表示本次有 LLM 智能待办；为 0 则本次为规则兜底，任务会偏模板化。
3. **多次生成**
   - 换店铺或换数据后多次点击「智能生成」，观察标题与描述是否在**维度**上明显区分（时段/互动/选品/引流等），而不再是一串雷同的「提升转化率」「增加时长」等。

## 若仍无区别

- 确认当前使用的**模型/智能体**是否擅长按指令做多维度区分（部分通用对话模型易忽略结构化要求）。
- 考虑在 system 中增加 **1～2 条示例**（示例中 2～3 条任务分属不同维度），以 few-shot 引导输出差异化。
