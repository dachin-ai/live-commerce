# 待办生成逻辑 - 11 号版本回顾

**说明**：本文档回顾截至 **2026-02-11** 的待办生成逻辑（“11 号最后一个版本”），便于与后续改动（如 Coze 工具入参、按日明细全量入参等）对照。**当前入参/出参与流式约定以《Coze对接说明-待办与话术入参出参》为准。**

---

## 一、整体架构（三层）

```
1. LLM 智能待办（优先） → 2. 月同比/年同比待办（补充） → 3. 阶段待办（兜底）
```

- **第一层**：Coze/OpenAI 一次调用，基于店铺完整数据 + 按日明细生成 6～10 条，来源 `llm_intelligent`。
- **第二层**：规则引擎按自然日对比上月/上年同期，GMV/转化率/订单/观看/时均GMV/直播时长等维度，超过阈值则生成待办，来源 `mom_comparison` / `yoy_comparison`。
- **第三层**：LLM 返回 0 条时用阶段规则兜底（冷启动/成长/成熟），来源 `stage`。

---

## 二、入口与主流程

| 项目 | 说明 |
|------|------|
| **用户操作** | 店铺管理 → 选店铺 → 点击「智能生成」 |
| **接口** | `POST /api/ai/generate-tasks`，body 含 `storeId`，需登录 |
| **核心函数** | `generateSuggestedTodosForStore(storeId, options)`（仅返回列表，不写库） |
| **写库与返回** | 路由内先清空该店铺下未完成待办，再调用上述函数，将结果去重/节日合并/条数上限后写入 `tasks` 表，返回前端 |

---

## 三、generateSuggestedTodosForStore 内部逻辑（11 号版本）

### 1. 前置准备（所有分支共用）

- 查店铺信息、类目、**已有待办标题**（pending）。
- 计算 `storeInfo`、`existingTaskTitles`、去重用 `existingTitleSet` / `existingPrefixSet`。
- 若「点击当天往前 30 天」有数据：`hasDataInStrict30 === true`，进入**分支 1**；否则视是否有「按日明细 + 汇总」入参进入**分支 2a/2b** 等。

### 2. 分支 1：有“当天往前 30 天”数据

1. **周期与数据**
   - 当前周期：`strictRange`（当天往前 30 天）；上一周期：`getPeriodDateRange(1)`。
   - 按日明细：`getRawDailyStatsForLLM(storeId, currentRange.dateFrom, dateTo)`，或 `options.rawDailyOverride`；当时实现为 **LIMIT 31**（最多约 30 天）。
   - 汇总：`currentStats`、`historicalStats`（上一周期）、`storeStage`、`timeContext`、`upcomingEvents` 等。

2. **第一步：LLM**
   - 调用 `generateIntelligentTodosWithLLM(...)`，传入店铺、类目、当前/历史统计、按日明细、已有待办、locale/countryCode 等。
   - 11 号版本要点：
     - **maxTokens**：文档《LLM输出压缩问题排查》建议从 1500 提到 **3000**，避免 Coze 压缩描述或减少条数。
     - **超时**：callLLMOnce 传 `timeoutMs: 120000`（2 分钟）；**重试**：最多 2 次，首次返回空则 1.5 秒后重试。
     - 解析：从 LLM 返回文本中 `extractTasksJsonFromText` 提取 `{"tasks":[...]}`，支持多种格式；解析后打 `source: 'llm_intelligent'`。

3. **第二步：LLM 返回 0 条时的兜底**
   - 若 `llmResult1.tasks.length === 0`：用 `generateStageBasedTasks(...)` 生成阶段待办，并记 `llmEmptyReason` 供前端/日志排查。

4. **第三步：月同比/年同比**
   - `getYearOverYearStats`、`getMonthOverMonthStats` 取去年同期、上月同期数据。
   - `generateComparisonTasks(currentStats, yoyStats, momStats, storeInfo)` 按 6 个维度（GMV、转化率、订单、观看、时均GMV、直播时长）与阈值生成待办，并入上一步结果。

5. **去重与打标**
   - `dedup(tasks)`：与已有待办按标题/前缀去重；并为任务打 `assignedRole`、`aiFeature`（`autoTagTaskRoleAndTool`）。
   - 结果写入 `result` 返回。

### 3. 分支 2a：无 30 天 DB 数据，但有 rawDailyOverride + metricsOverride

- 仅用入参拼 `currentStats`，无历史周期；仍调 `generateIntelligentTodosWithLLM`（无按日明细则无 rawDataBlock）；LLM 返回 0 条则阶段兜底；**不做**月同比/年同比（无历史数据）。

### 4. 分支 2b 等：无 30 天数据且无 Override

- 用「系统内已有最近 30 天」等逻辑尝试取 `existingRange`、`existingAgg`，若仍无数据则只返回「请上传最近30天的运营数据」等阶段兜底；若有数据则同样走「LLM → 阶段兜底 → 同比」的类似流程（多子分支，逻辑与分支 1 一致）。

## 四、generateIntelligentTodosWithLLM 入参（11 号版本）

- **店铺**：名称、平台、国家/区域、类目、店铺属性（目标人群、品牌定位、价格区间等）。
- **核心指标（最近 30 天）**：订单、观看、GMV、转化率、直播总时长、时均 GMV。
- **历史对比**：前期平均 GMV、平均转化及本期变化百分比（或无历史说明）。
- **按日明细**：`rawDailyStatsText` 放在【⭐ 按日明细数据】代码块中（当时为「最近 30 天」、LIMIT 31）。
- **业务上下文**：已有待办（避免重复）、用户补充说明。
- **语言**：locale、countryCode，用于要求 LLM 按指定语言输出。

**输出**：`{ tasks: IntelligentTodoItem[], llmEmptyReason?: string }`；每条含 title、description、priority，支持 title/task/name/content 等多种字段名。

## 五、11 号版本与后续改动的差异（简要）

| 项目 | 11 号版本 | 后续改动 |
|------|------------|----------|
| 按日明细条数 | `getRawDailyStatsForLLM` 使用 **LIMIT 31** | 去掉 LIMIT，**指定日期内全部数据明细**入参 |
| Coze 工具入参 | 无单独「工具参数」块 | 增加【供 Coze 工具 generate_todo_tasks 使用的参数】，含 **store_data** JSON、**raw_daily_table** 全文、platform、country、existing_tasks、store_attributes |
| store_data | 无 | 与 Coze 工具一致：conversion_rate、daily_views、avg_order_value、daily_orders、live_duration_hours、interaction_frequency_minutes、best_day、worst_day |
| Coze 流式解析 | 仅 extractCozeContent / 话术 tool_response | 支持 **tool_response** 中 `parsed.tasks` 数组，直接作为待办 JSON；并增强 **message_end** 的 content 提取 |
| 前端超时 | 60 秒 | 90 秒 |

**总结**：11 号最后一版待办生成 = **三层架构（LLM 优先 → 同比规则 → 阶段兜底）** + **单次 LLM 调用（最多 2 次重试、120s 超时、maxTokens 建议 3000）** + **按日明细当时为 LIMIT 31** + **无 Coze 工具专用 store_data/raw_daily_table 块**。后续在此基础上增加了「指定日期内全部明细」、Coze `generate_todo_tasks` 对齐入参及 tool_response 解析等。
