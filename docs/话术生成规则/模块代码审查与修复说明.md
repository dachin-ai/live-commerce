# 话术生成模块 - 代码审查与修复说明

对「话术生成」整模块（不限于 LLM 调用）做了通读与部分重构，结论与修改如下。

---

## 一、模块范围与调用链

| 层级 | 文件 | 职责 |
|------|------|------|
| 前端 | `frontend/src/components/AIFeatures.tsx` | 表单（产品名/价格/人群/话术类型/语言/促销）、生成按钮、流式结果展示、`relevanceWarning` 提示 |
| 前端 | `frontend/src/services/ai.ts` | `generateScriptStream()`：POST `/api/ai/script/stream`，传 `scriptType` 等，解析 SSE |
| 路由 | `backend/src/routes/ai-refactored.ts` | `POST /script`（同步，仅模板）、`POST /script/stream`（流式 LLM + 校验 + 模板兜底）；权限校验；公共解析 `parseScriptRequestBody()` |
| 市调与提示 | `backend/src/rules/scriptResearch.ts` | `runScriptResearch`、`buildSummaryForLLM`、`buildLLMSystemPrompt`、`buildLLMUserMessage`、`getPromptConfigForScriptType`（按话术类型配置） |
| 模板综合 | `backend/src/rules/scriptSynthesis.ts` | `synthesizeScript`：调用 `buildScriptContent` + 合规检查 + 店铺头 |
| 模板内容 | `backend/src/rules/scriptPrompts.ts` | `buildScriptContent(scriptType, language, params)`：中/英/泰各类型模板；`full-sales` 走 `fullSalesScript.ts` |
| 类型与元数据 | `backend/src/rules/scriptGeneration.ts` | `ScriptType`、`SCRIPT_TYPE_META`（目标/场景/要素） |
| 人群解析 | `backend/src/rules/scriptAudienceAnalysis.ts` | `analyzeTargetAudience` → 称呼、痛点、label |
| 合规 | `backend/src/rules/platformCompliance.ts` | `getPlatformCompliance`、`checkPlatformCompliance`、平台别名 |
| 可视化 | `backend/src/rules/scriptVisualRules.ts` | `segmentForVisual`、`injectVisualSuggestionsIntoContent`、`getVisualLegend`（按类型与配置） |
| 配置加载 | `backend/src/rules/loadScriptRulesConfig.ts` | 品类实践、人群关键词、平台合规、可视化规则、默认话术用语等 JSON |
| LLM 调用 | `backend/src/services/scriptLLM.ts` | `streamScriptFromLLM`、Coze/OpenAI 兼容、Coze 首行「只输出话术」提醒 |
| LLM 配置 | `backend/src/services/scriptLLMConfig.ts` | URL/API Key/允许用户、缓存 |
| 输出校验 | `backend/src/services/scriptOutputValidator.ts` | `isLikelyScriptContent`、跑题时模板兜底与 `relevanceWarning` |

---

## 二、已修复/优化项

1. **前端语言选项**  
   - 话术语言下拉补全 **泰语（th-TH）**，与后端及 `scriptPrompts` 的泰语模板一致。

2. **路由重复逻辑**  
   - `POST /script` 与 `POST /script/stream` 中原先各有一段相同的「body 解析 + 店铺查询 + storeContext + userInput」逻辑。  
   - 已抽取为 **`parseScriptRequestBody(req.body)`**，两处共用，减少重复与后续改动遗漏。

3. **人群默认配置兜底**  
   - `scriptAudienceAnalysis.ts` 中，从配置读取 `defaultAudience` 时若缺少 `label` 会变成 `undefined`。  
   - 已改为对 `addressTerm` / `painPointsHint` / `label` 做兜底，缺失时使用内置 `DEFAULT_AUDIENCE` 对应字段。

---

## 三、审查结论（无问题或已由前期修改覆盖）

- **scriptType 传递**：前端下拉为 `interaction` | `scenario` | `promotion` | `closing` | `full-sales`，与后端校验及 `SCRIPT_TYPE_META` 一致。  
- **模板与类型**：`scriptPrompts.buildScriptContent` 对五类 + 三种语言 + `full-sales` 走 `fullSalesScript`，分支完整。  
- **合规**：`resolvePlatform('直播')` 等未在别名表中的会落到 `default` 规则，行为合理。  
- **可视化**：`segmentForVisual` / `injectVisualSuggestionsIntoContent` 按 `scriptType` 与配置的 `startMarker` 工作；LLM 返回内容若无这些标记会整段成「正文」，属预期。  
- **流式与跑题**：LLM 返回报告类内容时由 `scriptOutputValidator` 判定并切换模板 + 设置 `relevanceWarning`，前端已展示该提示。

---

## 四、配置与扩展

- 品类实践、人群关键词、平台合规、可视化分段与演示动作、默认话术用语等均支持通过 **`backend/config/script-rules/*.json`** 覆盖或扩展，无需改代码即可调优。  
- 各话术类型的提示词与规则说明见同目录下各「话术类型」文档及 `README.md`。
