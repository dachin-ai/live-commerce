# 话术生成规则 - 测试检查清单

规则已接入：前端选择的话术类型会传入后端，`scriptResearch.ts` 按类型从 `SCRIPT_TYPE_META` 取目标/场景/要素，并写入系统提示与用户消息，流式接口 `/api/ai/script/stream` 使用该提示调用 LLM。可按下表逐一验证。

---

## 调用链确认（已生效）

| 环节 | 位置 | 说明 |
|------|------|------|
| 前端选择类型 | `AIFeatures.tsx` | 下拉框 `scriptForm.scriptType` 为 `interaction` / `scenario` / `promotion` / `closing` / `full-sales` |
| 请求体 | `ai.ts` → `body.scriptType` | 流式请求 POST `/api/ai/script/stream` 携带 `scriptType` |
| 后端接收 | `ai-refactored.ts` | `req.body.scriptType`，默认 `full-sales`，校验为五类之一 |
| 市调 | `scriptResearch.ts` → `runScriptResearch()` | 产出 `research`，含 `userInput.scriptType` |
| 系统提示 | `buildLLMSystemPrompt(research)` | 根据 `userInput.scriptType` 注入「当前生成「XX话术」。目标：…。场景：…。须包含要素：…」 |
| 用户消息 | `buildLLMUserMessage(research)` | 任务行含类型名、产品、目标、要素列表；随后市调摘要 + 结尾 |
| 调用 LLM | `streamScriptFromLLM({ systemPrompt, userMessage })` | 使用上述两段提示调用配置的 LLM/Coze |

## 建议逐类测试项

测试前请确保：已选店铺、已填产品名称、LLM 已配置且当前用户有权限。

| 话术类型 | 界面选择 | 建议自检点 |
|----------|----------|------------|
| **人群互动话术** | 人群互动话术 | 产出是否以互动为主：问句、扣1/评论/点赞、称呼、明确行动召唤；是否无长段独白、无策略报告 |
| **场景化塑品话术** | 场景化塑品话术 | 是否有场景/痛点再带出卖点、有画面感或小故事；是否避免干巴巴罗列卖点 |
| **促销活动话术** | 促销活动话术 | 是否包含价格/折扣/赠品/限时感、行动召唤；若填了促销信息是否被用上；数字是否泛化 |
| **逼单技巧话术** | 逼单技巧话术 | 是否有紧迫感、明确 CTA、简单选择；是否短而有力、无分析/报告 |
| **完整销售流程话术** | 完整销售流程话术（推荐） | 是否覆盖圈人群→塑品→打消顾虑→利益点→售后→逼单；结构是否清晰、有过渡 |

---

## 若某类型不符合预期

1. **确认请求带上了类型**：浏览器开发者工具 → Network → 找到 `script/stream` 请求 → Payload 中查看 `scriptType` 是否为所选值。
2. **查看文档对照提示词**：打开 `docs/话术生成规则/` 下对应文档，对照「系统提示词」「用户消息模板」与当前逻辑是否一致（逻辑在 `scriptResearch.ts` 的 `buildLLMSystemPrompt` / `buildLLMUserMessage`）。
3. **LLM 行为**：若接入的为第三方 bot（如 Coze），其自身人设过强可能弱化我们的提示，可尝试在 Coze 中弱化或调整 bot 指令，或换用更听话的模型接口。
